# 项目规划

本文档是智慧云脑诊疗平台当前唯一主规划文档，用于统一记录项目定位、业务主线、技术路线、三端边界、当前进度、真实差距、阶段计划和每次开发进展。后续答辩、开发、联调、演示和文档说明均以本文档为准。

更新时间：2026-07-06。

## 1. 项目定位

项目名称：智慧云脑诊疗平台。

项目定位：面向门诊场景的 AI 赋能 HIS / 无人 AI 医院实训项目，不涉及住院业务。

当前固定主线为 **脑科 / 神经外科门诊**，围绕“头痛、恶心、视物模糊、疑似脑部占位”场景，重点展示以下能力：

- AI 问诊与分诊
- AI 推荐医生
- 预约挂号、排班、候诊、叫号
- 检查 / 检验开立与结果回传
- 脑部 CT / MRI 医学影像辅助分析
- 医生审核 AI 建议后生效
- 处方、缴费、发药、电子病历归档

## 2. 产品结构与边界

### 2.1 三端独立方向

项目从当前开始，按 **患者端、医生端、管理员端三端独立** 的方向推进。

这里的“独立”指：

- 各自有独立登录界面
- 各自有独立首页 / 工作台
- 各自有独立主流程
- 各自有独立路由入口和权限边界
- 各自的视觉和交互以角色任务为中心，不再共用一个“切换角色后继续操作”的体验

这里的“独立”暂时 **不等于三个独立仓库**。当前更合理的落地方式是：

- 保持一个 `frontend/` 工程
- 继续共用请求层、类型、主题变量、基础组件
- 在同一个前端工程内拆成三个子应用命名空间：
  - `/patient/*`
  - `/doctor/*`
  - `/admin/*`

这样可以保留当前开发效率，同时把产品体验和权限边界拆清楚。

### 2.2 三端职责

#### 患者端

定位：移动端优先，承载患者自助就诊全流程。

主流程：

```text
登录 / 注册建档
-> 患者首页
-> AI 问诊或按科室挂号
-> 医生推荐
-> 选择日期 / 午别 / 时间
-> 线上挂号
-> 支付
-> 候诊状态
-> 历史挂号 / 个人中心
```

#### 医生端

定位：桌面端优先，承载医生接诊、审核、病历与检查处置流程。

主流程：

```text
医生登录
-> 医生工作台
-> 今日候诊队列
-> 接诊
-> 查看 AI 问诊摘要 / AI 建议
-> 医生确认病历
-> 开立检查检验 / 查看结果
-> 诊断与处方确认
```

#### 管理员端

定位：桌面端优先，承载医院基础资料、排班、审批和 AI 审计流程。

主流程：

```text
管理员登录
-> 管理控制台
-> 医生 / 科室 / 班次管理
-> 排班生成与审批
-> 规则干预
-> AI 审计与运行监控
```

### 2.3 当前接受的产品边界

- 登录注册先按三端独立登录页推进，但权限体系可以分阶段完善，不要求第一轮就做完整统一认证中心。
- 患者端继续优先接真实接口，不做大面积假业务。
- 医生端和管理员端在真实接口不足的模块可先做“真实数据优先 + 缺口留白”的工作台，不强造完整闭环。
- 支付继续使用后端模拟支付，不包装成真实第三方支付网关。
- 医学影像先展示图片路径、结果描述、AI 概率和报告字段，不实现完整 DICOM 工作站。
- 患者端不展示后端内部科室编码和底层细粒度号段。

## 3. 答辩与产品要求

### 3.1 AI 审核要求

答辩主线必须体现 **AI 给建议，医生审核后生效**。

需要区分两类数据：

- AI 产生的信息：AI 问诊摘要、分诊建议、检查建议、影像建议、诊断建议、处方建议
- 人工确认的信息：医生确认病历、最终检查单、最终诊断、最终处方

### 3.2 医学影像要求

- 聚焦脑部 CT / MRI
- 检查给图片，检验给数字报告
- 必须体现机器学习 / 深度学习模型能力
- 后续需要说明 PACS / DICOM 在系统中的位置
- 影像结果需要打通医生端、病历和诊断流程

## 4. 技术路线

- 后端：Python FastAPI 微服务
- AI 编排：LangChain / LangGraph
- 数据库：PostgreSQL + pgvector
- 基础设施：Redis、RabbitMQ、Nacos、Docker Compose
- 前端：Vue 3 + Vite + Vue Router + Pinia + Axios + Element Plus

前端结构原则：

- 一个前端工程，不拆成三个仓库
- 三套登录页、三套路由树、三套主布局、三套业务流程
- 共用 `api/`、公共类型、基础组件和样式变量
- 患者端移动端优先，医生端和管理员端桌面端优先

## 7. 当前真实差距

1. 三端虽然已有基础布局，且产品级全局角色切换入口已移除，但医生端和管理员端仍未形成真正独立的登录与会话边界。
2. 医生端和管理员端仍是占位页，尚未形成独立登录页和独立主流程。
3. 患者端已跑通第一阶段主链路，但报告查询、医院信息等仍缺少真实接口边界收口。
4. 演示数据仍需固化，避免临时生成排班影响联调和演示。
5. 三端分离后仍需要补三套会话状态和路由守卫，避免新的共享状态再次把入口耦合回去。

## 8. 三端独立后的实施方案

### 8.1 前端架构调整

前端不拆仓库，拆产品入口：

- 患者端入口：`/patient`
- 医生端入口：`/doctor/login`
- 管理员端入口：`/admin/login`

对应调整：

- 患者端保持移动端视觉体系
- 医生端建立独立登录页和工作台路由组
- 管理员端建立独立登录页和控制台路由组
- 已移除全局角色切换按钮，不再保留共享产品入口

### 8.2 鉴权与状态调整

建议从当前单一全局状态演进为：

- `patientSession`
- `doctorSession`
- `adminSession`

要求：

- 三端分别存储登录态、用户资料、权限标记和默认落地页
- 路由守卫按端校验，不允许跨端直接复用会话
- 未登录访问业务页时，重定向回对应登录页，而不是跳去别的端

### 8.3 路由分组调整

建议最终统一为：

```text
/patient
/patient/register
/patient/home
/patient/...

/doctor/login
/doctor/workbench
/doctor/queue
/doctor/encounter/:id
/doctor/records

/admin/login
/admin/console
/admin/schedules
/admin/doctors
/admin/audit
```

说明：

- 患者端现阶段可继续保留 `/patient` 作为登录页，避免打断已完成页面
- 医生端和管理员端从一开始就使用 `/doctor/login`、`/admin/login`
- 后续如果要统一风格，可以再把患者端平滑迁移到 `/patient/login`

### 8.4 接下来四个执行切片

后续不建议同时铺开三端全部页面，而应按“先补结构边界，再补医生端闭环，再补管理员端控制台，最后补演示与答辩收口”的顺序推进。

#### 切片 A：患者端收口 + 三端会话基础设施

目标：

- 保持患者端现有链路可用
- 为医生端、管理员端独立登录准备会话与路由守卫基础
- 不再让患者流程状态承担登录态职责

完成标准：

- 新增按端拆分的 session store 方案
- 路由层具备按端跳转登录页的能力
- 患者端现有页面在重构后不退化
- `frontend/src/router/index.ts` 完成按端路由入口整理

#### 切片 B：医生端独立登录与工作台首页

目标：

- 建立真正可进入的 `/doctor/login`
- 将当前医生端占位页升级为最小工作台首页
- 用真实挂号/候诊数据替代纯静态占位块

完成标准：

- 新增医生登录页
- 新增 `/doctor/workbench`
- 登录后可进入工作台
- 工作台至少显示真实候诊列表或真实挂号队列

#### 切片 C：医生端接诊详情最小闭环

目标：

- 让“AI 给建议，医生审核后生效”在医生端真正落地一段主流程

完成标准：

- 新增 `/doctor/encounter/:registerId`
- 能查看患者基本信息、AI 问诊摘要、病历草稿或诊疗建议
- 医生可执行至少一种“人工确认后生效”的动作

#### 切片 D：管理员端控制台 + 演示脚本收口

目标：

- 让管理员端承担排班、审批、审计的展示职责
- 同时固化一条稳定演示路径，避免每次临时拼装

完成标准：

- 新增 `/admin/login`、`/admin/console`
- 至少落地排班、审批、审计三个模块入口
- 补齐演示数据脚本或演示说明
- 输出一条可重复演示路线
