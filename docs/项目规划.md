# 项目规划

更新时间：2026-07-10。

本文档是 SmartBrainClinic 的总体状态与路线图唯一来源。前端页面和路由细节维护在 [frontend-plan.md](./frontend-plan.md)，仓库导航维护在 [项目结构说明.md](./项目结构说明.md)。

阶段回顾、问题记录和专项方案作为历史与报告留档保留：它们记录当时的决策、问题和验收，不用于覆盖本文档中的当前状态。

状态口径：`已提交` 为当前 Git 分支已有内容；`本地待合并` 为当前工作区实现，尚未随本次文档提交发布；`待验证` 为代码或方案存在但尚未通过真实数据、浏览器或联调验收的项目。

## 1. 项目目标

SmartBrainClinic 是面向门诊流程的 AI 赋能 HIS / 无人医院实训平台。当前固定演示主线为脑科 / 神经外科门诊，围绕头痛、恶心、视物模糊和疑似脑部占位等场景，形成以下闭环：

```text
患者建档 -> AI 分诊 / 手动选科 -> 医生推荐 -> 挂号支付 -> 候诊
-> 医生接诊 -> 检查 / 检验 / 处置 -> 病历确认 -> 处方 / 收费 -> 归档
```

核心原则：

- 真实接口优先，不用大面积假数据填充未完成页面。
- AI 只提供建议，最终诊疗动作由医生确认。
- 患者端、医生端、管理员端职责和会话边界独立。
- 先完成可演示主链，再扩展运营、审计和高级 AI 能力。

## 2. 技术与架构基线

- 后端：FastAPI 微服务，Gateway + Auth + Patient + Medical + Pharmacy + Billing。
- 前端：单一 Vue 3 工程，通过 `/patient/*`、`/doctor/*`、`/admin/*` 组织三端。
- 数据库：PostgreSQL + pgvector；Redis、RabbitMQ、Nacos 提供缓存、消息和服务发现能力。
- AI：OpenAI-compatible API、LangChain / LangGraph、向量检索和脱敏审计。
- 事务与稳定性：Outbox、幂等记录、收费防重锁和退费 Saga 已有基础实现。

## 3. 当前实现状态

### 3.1 患者端

已完成：

- 首页优先入口、登录查询和注册建档。
- AI 分诊、按科室选科、医生推荐、排班和号源选择。
- 线上预挂号、模拟支付、候诊状态和挂号历史。
- 个人中心、共享流程头部、底部导航和移动端主视觉。
- 本地待合并：挂号历史 Pinia 共享缓存，采用 `sessionStorage`、60 秒 TTL、并发请求复用和新挂号 / 支付后的主动失效。

当前边界：

- 报告查询、消息中心和医院信息没有真实业务闭环，不扩大假页面。
- 首次挂号历史请求仍受 Gateway -> Patient -> Auth 字典查询链路影响；共享 HTTP 连接池只能降低重复建连成本。

### 3.2 医生端

已完成：

- 独立医生登录、会话守卫和真实医生目录。
- 今日候诊队列、叫下一位、开始接诊和继续接诊。
- 工作台顶部挂号状态环图、分时段条形图、15 秒轮询、刷新时间及加载 / 空 / 错误状态。
- 接诊详情、挂号摘要、AI 病历草稿读取与医生确认。
- 相似病历召回、AI 医生助手。
- 检查、检验、处置三类真实开单和已开项目回看。

当前边界：

- 处方建议已并回接诊页：确认病历后才可按需生成，医生可调整用法、数量或移除药品，并通过单独确认动作创建正式处方；本地演示挂号已完成桌面及 `1024 x 768`、`390 x 844` 窄屏真实浏览器回归，仍待将该链路固化为可重复脚本。
- 已规划 AI 检查检验建议：后续只基于病历、AI 分诊上下文和已开项目生成候选检查或检验及其理由，不直接创建订单；医生筛选后复用统一开单签署。
- 工作台非空图表已完成代码实现，但当前本地 8 位医生今日队列均为空，真实非空样本仍需补验。
- 医生端接诊页已只读展示本次 AI 分诊摘要、患者提供的关键问答、性别年龄快照和来源说明；不自动改写病历或诊断。
- 医生端接诊页已按“左侧医生工作区、右侧 AI 支持区”调整，布局蓝图见 [前端计划](./frontend-plan.md#布局蓝图)；真实已登录接诊页的视觉回归仍待补验。

### 3.3 管理员端

已完成：

- `/admin/login`、`/admin/console`、独立 session 和路由守卫骨架。

尚未完成：

- 真实管理员鉴权。
- 医生 / 科室管理、排班生成、排班审批、规则干预和 AI 审计页面。
- 管理操作的权限、审计和失败恢复。

### 3.4 AI 分诊与病历上下文

当前工作区已实现：

- `ai_conversation_session` / `ai_conversation_message` 共用会话模型和迁移。
- `POST /api/v1/patient/triage` 支持 `patient_uuid`、`session_uuid` 多轮续聊并返回稳定会话标识；已自动读取患者性别和年龄，写入会话快照并注入真实 LLM 分诊上下文。
- 预挂号支持 `triage_session_uuid`，挂号后绑定 `register_uuid`。
- `GET /api/v1/patient/register/{register_uuid}/ai-context` 提供摘要、结构化结果、档案快照和消息。
- Medical 草稿消费者优先读取 `ai-context`，不存在时回退到 `register.symptoms`。

仍需完成：

- 将历史已确认病历并入医生侧草稿上下文。
- 结构化 `decision-support` 风险提醒、补问和检查建议接口。

### 3.5 后端与性能

- 本地待合并：Gateway 和微服务内部调用复用进程级共享 `httpx.AsyncClient`，各服务退出时关闭连接池。
- 当前测试覆盖患者辅助流程、AI 会话、Medical 辅助流程、收费退费保护和医生 Agent 工具等关键区域。
- 数据库迁移与实例状态必须分开判断；代码中存在迁移不代表目标数据库已经执行。

## 4. 当前差距与风险

| 优先级 | 差距 / 风险 | 处理原则 |
| --- | --- | --- |
| P0 | 仓库或默认配置中不得保留真实 LLM Key、Token、密码 | 使用 `.env`，发现泄露立即轮换 |
| P0 | AI 会话迁移未执行时，本地待合并的新链路无法完整工作 | 启动前检查 migration 状态 |
| P1 | 管理员端只有演示骨架 | 先接真实鉴权，再做排班和审计 |
| P1 | 医生端处方尚缺可重复自动化回归 | 将已通过的桌面及窄屏“建议、调整、确认开立”链路固化为可切换 viewport 的浏览器脚本 |
| P1 | 影像能力仍含 Mock 推理 | 界面和答辩明确标识边界 |
| P2 | 挂号历史首次请求可能仍慢 | 本地缓存已降低重复请求；仍需测真实耗时，再评估批量字典查询 |
| P2 | 非空医生队列缺当前样本 | 有真实当日挂号后补浏览器验收 |

## 5. 后续路线

### 阶段 A：医生端主链继续收口

1. 将已通过的“处方建议 -> 医生调整 -> 确认开立”桌面及窄屏链路固化为可重复浏览器回归脚本。
2. 实现 AI 检查检验建议：只生成候选项目和理由，医生选择、删除或调整后，复用统一开单确认；不得自动创建检查或检验单。
3. 用真实非空队列补验状态环图、时段图和开始接诊后的同步变化。
4. 固化“登录 -> 候诊 -> 接诊 -> 开单 -> 病历确认 -> 处方确认”的浏览器回归脚本。

### 阶段 B：管理员端真实闭环

1. 接入真实管理员鉴权和权限边界。
2. 接入科室、医生、排班、审批和规则干预。
3. 接入 AI 审计日志，只展示脱敏摘要和来源信息。

### 阶段 C：AI 上下文增强

1. 将历史已确认病历并入医生 AI 上下文。
2. 增加结构化决策辅助，不允许 AI 直接写入正式诊断或处置。

### 阶段 D：演示与交付

1. 固化可重复演示数据和启动检查。
2. 完成患者端 -> 医生端 -> 管理员端全链回归。
3. 整理答辩说明、架构图和 AI 边界说明。

## 6. 验证基线

每个里程碑至少完成与改动范围相符的验证：

- 前端：`npm run build`，并对关键路由做桌面 / 窄屏浏览器检查。
- 后端：`python -m py_compile` 或相关 `pytest`。
- 数据库：核对迁移已执行，不只检查 SQL 文件存在。
- AI：检查 source、model、fallback 和“医生确认后生效”边界。
- 文档：检查链接、路由、当前状态和下一步与代码一致。

## 7. 文档维护规则

- `README.md`：启动方式、技术基线和文档入口。
- 本文档：总体状态、差距、优先级和阶段路线。
- `frontend-plan.md`：前端路由、页面、状态和执行切片。
- `项目结构说明.md`：仓库与代码导航。
- 数据库、演示数据、GitHub 协作保留独立参考文档。
- 完成专项后把当前结论并回主文档；专项设计、阶段回顾和问题复盘保留为历史记录，并在标题或开头标注日期和适用范围。
- `当前项目回顾.md` 用于阶段快照，`问题记录.md` 使用“现象、原因、已处理、后续注意”模板，便于后续报告引用。
- Git 历史记录代码变更；文档记录关键决策、验证和问题复盘，避免每日状态散落在多个主计划中。
