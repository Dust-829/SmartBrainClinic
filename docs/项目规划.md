# 项目规划

本文档以当前仓库真实实现为基线。项目后端已确定采用 Python FastAPI 微服务；前端尚未初始化；早期 Java/Spring 目录和表述仅作为历史规划参考。

更完整的需求、进度和实施计划见 [实训需求与项目实施计划](实训需求与项目实施计划.md)。

## 当前状态

已具备：

- FastAPI 微服务：Gateway、Auth、Patient、Medical、Pharmacy、Billing。
- PostgreSQL + pgvector。
- RabbitMQ Outbox 消息机制。
- Nacos 服务发现 fallback。
- AI 分诊、医生推荐、AI 病历草稿、AI 处方推荐、AI 排班原型。
- 挂号、排班、检查/检验、缴费、处方、发药等后端接口。

未完成：

- 前端患者端、医生端、管理员端。
- 医生端左 HIS / 右 AI 双屏界面。
- 真实 DICOM/PACS 和深度学习影像模型。
- 统一的 AI 建议与医生确认数据规范。
- 登录、角色权限、审计日志。
- 自动化测试和一键演示脚本。
- 配置安全整改。

## 近期主线

当前开发主线固定为神经外科/脑科门诊闭环：

```text
张三注册 -> AI 问诊 -> AI 分诊 -> 推荐医生 -> 挂号缴费
-> 华佗接诊 -> AI 病历草稿 -> 检查/检验建议
-> 脑部 CT/MRI AI 影像识别 -> 医生确认诊断
-> AI 处方建议 -> 医生确认处方 -> 财务缴费 -> 药房发药
```

## 阶段一：文档和演示数据统一

- 修订旧需求文档中的 Java/Spring 服务命名和端口。
- 明确最终技术路线为 Python FastAPI。
- 固化张三、华佗、神经外科、脑部 CT/MRI 的演示剧本。
- 补 DICOM/PACS、AI 建议审核、处方/医嘱关系说明。
- 准备科室、医生、排班、药品、医技项目、患者、检验报告和影像样例数据。

## 阶段二：前端三端最小闭环

- 患者端：移动端布局，注册、AI 问诊、医生推荐、挂号、支付、候诊。
- 医生端：桌面双屏布局，候诊队列、病历确认、检查检验、影像结果、诊断、处方。
- 管理员端：排班规则、AI 排班审批、医生评价、统计看板。

## 阶段三：业务闭环联调

- 连续跑通建档、分诊、推荐医生、挂号、支付、候诊。
- 连续跑通接诊、病历草稿、检查检验、影像结果、诊断确认。
- 连续跑通处方推荐、医生确认、缴费、药房发药。
- 补充关键异常状态：退号、退费、退药、号源冲突。

## 阶段四：医学影像强化

- 明确 DICOM/PACS 数据流和影像保存方式。
- 用真实或半真实脑部 CT/MRI 示例替换纯 Mock。
- 将影像推理封装为可替换模型服务。
- 输出概率、报告、医生确认状态，并在医生端可视化。

## 阶段五：AI 加分能力

- AI 排班：自然语言规则转排班记录，管理员审批。
- AI 分诊台：按 CT 室负载、医生擅长和紧急程度调度。
- AI 药房：库存预警、补货建议、药品消耗趋势。
- AI 质量评估：病历质量、处方合理性、AI 建议采纳率。

## 阶段六：工程稳定性

- 移除硬编码密钥，补 `.env.example`。
- 修正 Docker/Nacos 注册地址。
- 统一 `bill` / `billing` 命名。
- 补登录、权限和审计。
- 补核心自动化测试：抢号、缴费退费、库存扣减、消息投递。
- 增加一键启动和一键演示脚本。
