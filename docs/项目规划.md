# 项目规划

本文档作为当前项目唯一的总规划文档，统一承载需求摘要、业务流程、当前进度、差距分析和后续计划。后续答辩、开发、联调、PPT 和演示脚本均以本文档为准。

更新时间：2026-06-30。

## 1. 项目定位

项目名称：智慧云脑诊疗平台。

项目定位：面向门诊场景的 AI 赋能 HIS / 无人 AI 医院实训项目，不涉及住院业务。

当前建议固定业务主线为 **脑科 / 神经外科门诊**，围绕“头痛、恶心、视物模糊、疑似脑部占位”场景，重点展示以下能力：

- AI 问诊与分诊
- AI 推荐 3 到 5 位医生
- 挂号、排班、候诊、叫号
- 检查检验开立与结果回传
- 脑部 CT / MRI 医学影像辅助分析
- 医生审核 AI 建议
- 处方、缴费、发药、电子病历归档

## 2. 技术路线

当前最终路线已经明确，不再按早期 Java 规划推进。

- 后端：Python FastAPI 微服务
- AI 编排：LangChain / LangGraph
- 数据库：PostgreSQL + pgvector
- 基础设施：Redis、RabbitMQ、Nacos、Docker Compose
- 前端：Vue 3 + Pinia + Axios + Element Plus

说明：Spring Boot / JDK17 / Maven 3.9+ 属于实训允许路线和历史规划背景，但本项目当前真实实现以后端 Python 为主。

## 3. 考核要求摘要

### 3.1 角色端要求

至少保证三个端，并且界面明显不同：

- 患者端：手机端优先，负责注册、AI 问诊、分诊、医生推荐、挂号、支付、候诊、结果查看
- 医生端：电脑端，左侧传统 HIS 业务区，右侧 AI 辅助区
- 管理员端：电脑端，负责排班、审批、统计、基础资料管理

可扩展端包括医技端、药房端、财务端。

### 3.2 业务闭环要求

项目需要体现完整门诊链路：

```text
张三注册
-> AI 问诊
-> AI 判断是否建议到院
-> AI 分诊推荐脑科 / 神经外科
-> AI 推荐 3 到 5 位当班医生
-> 患者选择医生并挂号
-> 线上或线下支付
-> 进入候诊队列并在时间段内报到
-> 医生叫号接诊
-> AI 生成病历草稿
-> 医生确认病历
-> AI 推荐检查 / 检验清单
-> 医生增删改并确认
-> 患者缴费
-> 医技端执行检查检验
-> 检查产生 CT / MRI 图片，检验产生数字报告
-> AI 输出影像辅助分析结果
-> 医生审核诊断建议
-> AI 推荐处方
-> 医生审核确认处方
-> 患者缴费
-> 药房发药
-> 电子病历归档
```

### 3.3 AI 审核要求

项目虽然强调“无人 AI 医院”，但当前答辩主线必须体现 **AI 给建议，医生审核后生效**。

必须区分两类数据：

- AI 产生的信息：AI 问诊摘要、分诊建议、检查建议、影像建议、诊断建议、处方建议
- 人工确认的信息：医生确认病历、最终检查单、最终诊断、最终处方

### 3.4 医学影像要求

- 聚焦脑部 CT / MRI
- 检查给图片，检验给数字报告
- 必须体现机器学习 / 深度学习模型
- 最好说明 PACS / DICOM 标准
- 影像结果需要打通医生端、病历和诊断流程

## 4. 当前项目结构

```text
SmartBrainClinic/
  backend/                      当前核心实现，FastAPI 微服务后端
    app/
      main.py                   Gateway 网关，端口 8000
      common/                   配置、数据库、MQ、Nacos、AI 审计等公共能力
      microservices/
        auth/                   基础资料 / 医生 / 科室服务，端口 8001
        patient/                患者 / 挂号 / 分诊 / 排班服务，端口 8002
        medical/                病历 / 检查 / 检验 / 影像服务，端口 8003
        pharmacy/               药房 / 处方 / 库存服务，端口 8004
        billing/                财务 / 缴费 / 退费服务，端口 8005
    docker-compose.yml          PostgreSQL、Redis、RabbitMQ、Nacos 编排
    migrations/                 数据库增量迁移脚本
    tests/                      后端测试
  frontend/                     Vue 3 前端工程，已完成阶段 0 初始化
  docs/                         项目文档
  outputs/                      输出成果，如分工表
  database/                     数据脚本目录
  scripts/                      辅助脚本目录
```

说明：`services/` 是早期 Java / Spring 占位目录，不是当前核心实现。

## 6. 当前真实差距

当前系统已经能启动，但还不能认定为“完整项目已完成”。主要差距如下：

1. 前端已完成阶段 0 骨架，但患者端、医生端、管理员端的真实业务页面还没有联调完成。
2. 业务接口虽然较完整，但还没有用固定演示数据跑通整条门诊主链路。
3. 医学影像仍然是 Mock 推理，不是真实深度学习模型。
4. PACS / DICOM 还没有形成清晰的数据流说明。
5. AI 建议和人工确认虽然已部分实现，但还需要统一数据规范。
6. 完整登录、角色权限、关键操作审计仍不足。
7. 药房、财务、检查、处方等模块还需要做串联验证。
8. 当前缺少固定的答辩剧本和演示数据集。

## 7. 当前开发原则

当前策略调整为前后端并行推进，但前端只接真实后端接口，不写大面积假业务。

优先级如下：

1. 患者端主链路联调
2. 固定演示数据和演示脚本
3. 医生端工作台联调
4. 管理员端基础后台联调
5. 医学影像与 AI 审核强化
6. 权限、测试、文档和答辩材料补全

原因很简单：前端已经完成工程骨架，继续停在空页面没有意义；但如果脱离真实接口单独做页面，后面返工成本会更高。
