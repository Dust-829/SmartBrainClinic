# 项目规划

本文档是智慧云脑诊疗平台当前唯一主规划文档，用于统一记录项目定位、业务主线、技术路线、当前进度、真实差距、阶段计划和每次开发进展。后续答辩、开发、联调、PPT 和演示脚本均以本文档为准。

更新时间：2026-06-30。

## 1. 项目定位

项目名称：智慧云脑诊疗平台。

项目定位：面向门诊场景的 AI 赋能 HIS / 无人 AI 医院实训项目，不涉及住院业务。

当前固定主线为 **脑科 / 神经外科门诊**，围绕“头痛、恶心、视物模糊、疑似脑部占位”场景，重点展示以下能力：

- AI 问诊与分诊
- AI 推荐医生
- 预约挂号、排班、候诊、叫号
- 检查 / 检验开立与结果回传
- 脑部 CT / MRI 医学影像辅助分析
- 医生审核 AI 建议后生效
- 处方、缴费、发药、电子病历归档

### 2.2 AI 审核要求

答辩主线必须体现 **AI 给建议，医生审核后生效**。

需要区分两类数据：

- AI 产生的信息：AI 问诊摘要、分诊建议、检查建议、影像建议、诊断建议、处方建议。
- 人工确认的信息：医生确认病历、最终检查单、最终诊断、最终处方。

### 2.3 医学影像要求

- 聚焦脑部 CT / MRI。
- 检查给图片，检验给数字报告。
- 必须体现机器学习 / 深度学习模型能力。
- 后续需要说明 PACS / DICOM 在系统中的位置。
- 影像结果需要打通医生端、病历和诊断流程。

### 2.4 当前可接受的产品边界

- 登录注册先做患者端轻量入口：通过已建档证件号查询患者，不等待完整 RBAC。
- 支付先做后端模拟支付，不包装成真实第三方支付网关。
- 医学影像先展示图片路径、结果描述、AI 概率和报告字段，不实现完整 DICOM 工作站。
- 管理员端先做排班管理、审批和 AI 审计，不先做复杂 BI。
- 患者端不展示后端内部科室编码和底层细粒度号段。

## 3. 主业务闭环

完整门诊链路如下：

```text
患者注册 / 登录
-> AI 问诊
-> AI 判断是否建议到院
-> AI 分诊推荐神经外科
-> AI 推荐有号源医生
-> 患者选择医生和门诊班次
-> 线上预挂号锁号
-> 模拟支付
-> 进入候诊队列并查看候诊状态
-> 医生叫号接诊
-> AI 生成病历草稿
-> 医生确认病历
-> AI 推荐检查 / 检验清单
-> 医生增删改并确认
-> 患者缴费
-> 医技端执行检查 / 检验
-> 检查产生 CT / MRI 图片，检验产生数字报告
-> AI 输出影像辅助分析结果
-> 医生审核诊断建议
-> AI 推荐处方
-> 医生审核确认处方
-> 患者缴费
-> 药房发药
-> 电子病历归档
```

当前演示建议固定：

- 患者：张三
- 科室：神经外科
- 症状：头痛、恶心、视物模糊
- 医生：优先使用当前数据库中有神经外科排班的医生
- 检查：脑部 CT / MRI
- 输出：影像辅助结果、检验数字报告、电子病历、处方、发药

## 4. 技术路线

当前最终路线已经明确，不再按早期 Java 规划推进。

- 后端：Python FastAPI 微服务
- AI 编排：LangChain / LangGraph
- 数据库：PostgreSQL + pgvector
- 基础设施：Redis、RabbitMQ、Nacos、Docker Compose
- 前端：Vue 3 + Vite + Vue Router + Pinia + Axios + Element Plus

说明：Spring Boot / JDK17 / Maven 3.9+ 属于实训允许路线和历史规划背景，但本项目当前真实实现以后端 Python 为主。

## 5. 项目结构

```text
SmartBrainClinic/
  backend/                      当前核心实现，FastAPI 微服务后端
    app/
      main.py                   Gateway 网关，端口 8000
      common/                   配置、数据库、MQ、Nacos、AI 审计等公共能力
      microservices/
        auth/                   基础资料 / 医生 / 科室服务，端口 8001
        patient/                患者 / 挂号 / 分诊 / 排班服务，端口 8002
        medical/                病历 / 检查 / 检验 / 影像服务，端口 8003
        pharmacy/               药房 / 处方 / 库存服务，端口 8004
        billing/                财务 / 缴费 / 退费服务，端口 8005
    docker-compose.yml          PostgreSQL、Redis、RabbitMQ、Nacos 编排
    migrations/                 数据库增量迁移脚本
    tests/                      后端测试
  frontend/                     Vue 3 前端工程
  docs/                         项目文档
  outputs/                      输出成果，如分工表
  database/                     数据脚本目录
  scripts/                      辅助脚本目录
```

说明：`services/` 是早期 Java / Spring 占位目录，不是当前核心实现。

## 7. 患者端交互设计规则


### 7.1 患者端首页与挂号入口规则

![患者端首页概念图](assets/patient-home-concept.png)

- `/patient/home` 是登录后的患者端首页，不再直接承担科室选择页的职责。
- `/patient/departments` 是按科室挂号入口，进入时弹窗询问是否需要智能分诊。
- 智能分诊不作为首页独立入口，只是按科室挂号流程里的辅助选科步骤。
- 首页功能先保留按科室挂号、候诊状态、历史挂号、就诊码、报告查询和医院信息，不堆叠未接真实接口的功能。

### 7.2 登录注册规则

- `/patient` 为患者端首页，优先展示登录界面。
- 登录使用姓名 + 证件号查询已建档患者。
- 登录成功后进入 `/patient/home`，继续问诊、推荐、挂号、支付、候诊主链路。
- 登录页提供“没有账号，去注册建档”入口。
- 注册成功后自动跳回登录页，并保留姓名和证件号。
- 如果注册时发现证件号已注册，也跳回登录页并保留输入信息，提示用户直接登录。
- 当前为轻量患者入口，不等同于完整 RBAC 账号体系。

### 7.3 AI 问诊与推荐规则

- AI 问诊结果必须显示 AI 来源：真实大模型或 mock 兜底。
- 当前后端未配置 `LLM_API_KEY` 时，分诊会走 `mock-triage`，页面必须避免误导为真实大模型。
- 患者端只展示科室中文名，不展示内部科室代码。
- 医生推荐必须来自真实接口和真实排班，不写本地假医生或假号源。

### 7.4 挂号时间选择规则

为贴近真实医院预约挂号体验，患者端不再一次性展示医生某天全部细粒度号段。底层 `scheduling_time_slot` 继续用于锁号和队列排序，但患者按“日期 -> 上午 / 下午 -> 具体时间”逐步选择。

前端交互规则：

- 医生推荐页展示最近可约班次摘要：日期、上午 / 下午、余号、挂号费。
- 挂号确认页按未来 7 天聚合展示班次卡片。
- 每天最多展示“上午门诊 08:00-12:00”和“下午门诊 13:00-17:00”两类；如后续支持夜诊，再增加“夜诊”。
- 卡片展示科室、医生、日期、午别、剩余号源、挂号费和建议到院时间。
- 不向患者展示 8 分钟 / 10 分钟等底层小号段。
- 患者选择具体时间后，系统使用该时间对应号段锁号。

后端与数据规则：

- 继续保留 `scheduling_time_slot` 作为并发锁号、队列排序、候诊计算的内部数据结构。
- 前端传患者实际选择的 `scheduling_time_slot_uuid`。
- 后续可优化为后端提供按 `scheduling_actual_id` 查询实时余号的接口，再由前端展示可选时间。
- 支付成功后在候诊页展示建议到院时间、前方等待人数、诊室位置和候诊状态。


### 7.5 患者端页面设计稿

![患者端页面设计总览](assets/patient-pages-concept-board.png)

- 患者端其他页面应延续首页风格，使用蓝白医疗小程序式界面。
- 其他患者端页面必须延续首页视觉风格，不再另起一套界面风格。
- 登录和注册建档保持轻量表单，注册成功或已注册后回到登录。
- 挂号入口先选择科室，弹窗提供智能分诊辅助，但不强制使用 AI。
- 医生推荐、挂号确认、支付和候诊页都应围绕真实接口返回的医生、号源、挂号单和队列状态展示。

## 8. 当前真实差距

当前系统已经能启动并进入患者端主链路开发，但还不能认定为完整项目完成。主要差距如下：

1. 患者端主链路页面已拆出，挂号确认页已按“日期 -> 午别 -> 具体时间”收敛号源展示；后续需补号源过期、余号刷新和无号状态。
2. 医生端真实接诊工作台还未联调完成。
3. 管理员端真实排班审批和审计页面还未联调完成。
4. 业务接口虽然较完整，但还缺少一套稳定、可重复的一键演示数据脚本。
5. AI 分诊当前在未配置 key 时走 mock 兜底，真实大模型接入状态需要明确配置和演示说明。
6. 医学影像仍然是 Mock 推理，不是真实深度学习模型。
7. PACS / DICOM 还没有形成清晰的数据流说明。
8. AI 建议和人工确认虽然已部分实现，但前端展示规范还需要统一。
9. 完整登录、角色权限、关键操作审计仍不足。
10. 药房、财务、检查、处方等模块还需要做串联验证。
11. 当前缺少固定答辩剧本、演示脚本和演示数据集。

## 9. 当前开发原则

- 前端只接真实后端接口，不写大面积本地假业务。
- 先跑通患者端主链路，再切医生端和管理员端。
- AI 结果必须标明来源，区分真实 LLM、规则、mock、fallback。
- 医生推荐必须依赖真实医生、真实科室、真实排班和真实号源。
- 患者端优先移动端体验，避免把内部编码和底层技术细节暴露给患者。
- 后端已有的细粒度数据结构可以保留，但前端要做合理的信息收敛。
- 任何临时演示数据都要沉淀为脚本或文档，避免只靠手工操作。

优先级：

1. 患者端主链路联调
2. 固定演示数据和演示脚本
3. 医生端工作台联调
4. 管理员端基础后台联调
5. 医学影像与 AI 审核强化
6. 权限、测试、文档和答辩材料补全

## 2026-06-30 挂号记录页设计补充

挂号记录页需要作为患者端独立页面规划，建议路由为 `/patient/registers`，底部导航第二项固定为“挂号记录”，不再承担“去挂号”的入口职责。去挂号仍由首页宫格“按科室挂号”或记录空状态中的“去按科室挂号”进入。

### 状态一：未登录

未登录时不请求历史挂号接口，也不显示“暂无挂号记录”。页面只展示登录引导：

- 标题：挂号记录
- 主提示：请先登录后查看挂号记录
- 说明：登录后可查看线上挂号、支付状态和候诊信息。
- 主按钮：去登录，跳转 `/patient`
- 次按钮：注册建档，跳转 `/patient/register`

### 状态二：已登录

已登录后使用当前患者 `patient.uuid` 调用真实接口：

```text
GET /api/v1/patient/{patient_uuid}/registers/detail
```

页面状态需要互斥展示：

- 加载中：显示挂号记录骨架屏。
- 有记录：按真实接口返回的挂号记录渲染列表。
- 无记录：显示“暂无挂号记录”，并提供“去按科室挂号”按钮。
- 接口失败：显示错误提示和“重新加载”按钮。

记录卡片字段优先使用真实返回值：

- 科室：`dept_name`
- 医生：`employee_name`
- 日期：`actual_schedule_date || visit_date`
- 时间：`actual_time_range || noon`
- 状态：`visit_state_str`
- 挂号费：`regist_money`
- 诊室：`clinic_room_name`
- 症状：`symptoms`

已登录页面不显示“请先登录”；未登录页面不显示任何挂号记录卡片。这两个状态不能混在同一个实际页面中。
