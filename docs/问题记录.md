# 问题记录

本文档记录 SmartBrainClinic 开发过程中已经遇到的问题、原因判断、处理状态和后续注意点。后续每次遇到会影响联调、演示、设计一致性或开发效率的问题，都应追加到这里。

更新时间：2026-07-02。

## 1. 前端请求 Network Error

### 现象

- 患者端注册按钮点击后显示 `Network Error`。
- 8000 网关恢复后，注册按钮仍然报 `Network Error`。

### 原因判断

- 第一轮问题与后端网关不可用有关。
- 网关恢复后仍报错，通常需要继续检查前端请求地址、浏览器控制台、网关代理、CORS、后端接口异常和请求 payload。
- 这类错误不能只看前端 toast，需要同时看浏览器 Network 面板和后端网关日志。

### 已处理

- 前端统一通过公共请求层接后端网关。
- 患者端注册建档继续使用真实接口 `POST /api/v1/patient`。

### 后续注意

- 遇到 `Network Error` 时优先确认：
  - `http://localhost:8000` 网关是否运行。
  - 前端 `VITE_API_BASE_URL` 是否指向网关。
  - 浏览器 Network 面板是否有预检请求失败、CORS 失败或连接拒绝。
  - 后端接口是否返回了非 JSON 异常。

## 2. AI 问诊结果展示内部科室代码

### 现象

- AI 问诊结果曾显示 `SJWK`，用户无法理解这个内部代码。

### 原因判断

- 后端分诊结果或科室对象中包含内部科室代码。
- 前端直接展示了接口字段，未转换为患者可理解的中文科室名。

### 已处理

- 患者端页面不再向患者展示 `SJWK` 等内部代码。
- 前端计划中明确：内部代码只用于接口参数，展示层必须使用中文科室名，例如“神经外科”。

### 后续注意

- 所有患者可见页面都不能展示后端内部编码。
- 可展示字段优先级：中文名称 > 业务描述 > 内部 code。

## 3. AI 是否真实接入不清晰

### 现象

- 用户询问“这是真的接入了 AI 吗”。
- 页面曾给人一种所有 AI 回复都来自真实大模型的感觉。

### 原因判断

- AI 分诊接口可能存在 mock、规则或 fallback 结果。
- 页面没有清楚展示 `source`、`model`、`confidence` 等来源信息。

### 已处理

- 前端计划中加入 AI 来源展示要求。
- 页面文案中保留“AI 回复仅供参考，最终诊疗由医生确认”。

### 后续注意

- AI 结果必须标明来源，至少区分：
  - 真实 LLM。
  - 规则引擎。
  - mock。
  - fallback。
- 答辩时不能把 mock 或 fallback 说成真实 AI 接入。

## 4. 医生推荐为空

### 现象

- AI 问诊后没有医生推荐。

### 原因判断

- 医生推荐依赖真实科室、医生和排班。
- 如果对应科室没有医生、医生没有有效排班，推荐结果会为空。

### 已处理

- 前端医生推荐页接真实 `POST /api/v1/patient/recommend-doctors`。
- 前端计划中明确：医生推荐必须来自真实医生、真实排班和真实号源。

### 后续注意

- 推荐为空时不要用假医生补页面。
- 应检查：
  - 科室是否存在。
  - 科室下是否有医生。
  - 医生是否有当前日期之后的排班。
  - 排班是否还有可用号源。

## 5. 登录注册流程不符合真实使用习惯

### 现象

- 初始患者页不是登录优先。
- 注册完成后曾返回登录页，流程较绕。
- 返回按钮缺失、返回目标不符合预期。
- 返回箭头显示为问号。

### 原因判断

- 页面拆分早期没有形成固定导航规则。
- 使用了不稳定字符或图标，导致箭头显示异常。
- 注册成功后的状态流转没有和患者端流程状态统一。

### 已处理

- 患者端首页优先登录。
- 登录页和注册页保留返回按钮，返回目标统一到患者端首页。
- 注册按钮文案简化为“注册”。
- 注册成功后直接进入已登录状态。
- 身份证输入自动匹配出生日期，并校验证件出生日期合法性。

### 后续注意

- 移动端返回按钮必须使用稳定图标或 CSS 绘制，避免乱码。
- 登录、注册、首页之间的跳转必须保持一致：
  - 首页 -> 登录。
  - 登录 -> 注册。
  - 注册成功 -> 进入患者已登录流程。

## 6. 挂号时间段过多

### 现象

- 医生可选挂号时间过多，患者界面显得混乱。
- 用户希望改为“选日期 -> 选上午 / 下午 -> 选时间”。
- 还要求显示每个时间段剩余号源。

### 原因判断

- 后端底层排班可能按细粒度 time slot 存储。
- 前端直接铺开所有 time slot 不符合医院预约挂号常见体验。

### 已处理

- 前端计划中明确挂号时间选择方案：
  - 先选日期。
  - 再选上午 / 下午。
  - 再选具体时间段。
  - 时间段数量收敛。
  - 每个时间段展示剩余号源。
- 挂号确认页已按该方向改造。

### 后续注意

- 患者端不展示全部底层 `scheduling_time_slot`。
- 前端选择具体时间后，仍要传真实 `scheduling_time_slot_uuid` 给线上挂号接口。
- 若后端缺少聚合余号字段，前端只能基于现有排班字段做临时聚合，后续应补后端聚合接口。

## 7. 智能分诊入口位置不合理

### 现象

- 首页曾有独立智能分诊入口。
- 用户指出智能分诊已经集成在按科室挂号流程中。

### 原因判断

- 智能分诊不是独立主业务，而是帮助患者选择科室的辅助步骤。

### 已处理

- 首页概念设计中移除独立“智能分诊”入口。
- 按科室挂号页进入时弹出智能分诊提示：
  - “暂不需要”：手动挂号。
  - “帮我”：进入 AI 问诊。

### 后续注意

- 首页只保留高频服务入口。
- 智能分诊应作为挂号流程中的可选辅助，不强制阻断患者手动挂号。

## 8. 患者端页面风格不统一

### 现象

- 登录、注册、首页、AI 问诊、个人中心、挂号记录等页面视觉风格一度不一致。
- 首页曾出现顶部假时间、WiFi 等状态栏。
- 部分文本和门诊号区域出现重叠。

### 原因判断

- 页面是逐步补齐的，早期缺少统一患者端视觉规范。
- 移动端卡片尺寸、文字长度和响应式约束不足。

### 已处理

- 患者端统一采用蓝白医疗移动端风格。
- 首页按参考图重做，移除假的时间 / WiFi / 电量状态栏。
- 文档中加入患者端页面设计总览和统一风格要求。

### 后续注意

- 患者端页面应保持：
  - 蓝色医疗渐变头部。
  - 白色圆角卡片。
  - 浅蓝提示条。
  - 圆形图标。
  - 小程序式底部导航。
- 所有中文按钮和卡片文字必须留足空间，避免重叠。

## 9. 底部导航第二项语义错误

### 现象

- 首页底部导航第二项一度是“挂号”。
- 用户要求第二项应查询患者挂号记录，而不是去挂号。

### 原因判断

- 底部导航与首页功能网格职责混淆。
- “去挂号”已经在首页功能入口和按科室挂号页中承担，不应重复占用底部栏。

### 已处理

- 底部导航第二项改为“挂号记录”。
- 新增独立路由 `/patient/registers`。
- 未登录态和已登录态分开：
  - 未登录：提示先登录，不请求接口。
  - 已登录：调用真实历史挂号接口。

### 后续注意

- 底部导航职责固定为：
  - 首页。
  - 挂号记录。
  - 消息。
  - 我的。
- 不要再把“挂号记录”指向个人中心 hash。

## 12. Markdown 中文显示乱码风险

### 现象

- PowerShell `Get-Content` 查看部分中文文档时出现乱码。

### 原因判断

- Windows 终端编码和文件 UTF-8 显示不一致。
- 文件本身可能是 UTF-8，但 PowerShell 输出显示异常。

### 已处理

- 使用 Node.js 按 UTF-8 读取时确认中文内容正常。
- 后续写文档尽量使用 UTF-8 无 BOM。

### 后续注意

- 判断文档是否真乱码时，不只看 PowerShell 显示。
- 可用 Node.js 或编辑器确认真实编码。

## 15. Windows sandbox 和文件写入问题

### 现象

- `apply_patch` 偶发失败，提示 `windows sandbox failed: helper_unknown_error`。
- 普通 PowerShell 写文件有时出现权限问题。
- `npm run build` 在普通沙箱下可能因为写入 `vite.config.ts.timestamp-*.mjs` 报 `EPERM`。

### 原因判断

- Codex Windows 沙箱对某些文件读取、写入和临时文件生成支持不稳定。
- Vite 构建需要在项目目录生成临时配置文件。

### 已处理

- 必要时使用授权 PowerShell 做定向替换或写入。
- `npm run build` 在授权后可通过。

### 后续注意

- 能用 `apply_patch` 时优先用。
- 遇到 sandbox helper 错误时，不要反复重试同一失败路径，可改用更小范围的 PowerShell 定向写入。
- 构建失败如果是 Vite 临时文件 `EPERM`，可授权重跑同一构建命令。

## 16. PowerShell 脚本细节问题

### 现象

- PowerShell 启动时用户 profile 可能因为执行策略报错。
- `$HOME` 是只读变量，脚本中使用 `$home` 变量会冲突。
- 字符串替换时双引号和单引号嵌套容易导致解析错误。

### 原因判断

- Windows PowerShell 变量大小写不敏感。
- 脚本执行策略会影响 profile 加载。

### 已处理

- 部分命令使用 `login: false` 绕开 profile。
- 避免使用 `$home` 作为普通变量名。
- 对复杂替换使用 `.NET File.ReadAllText / WriteAllText` 和更稳定的字符串写法。

### 后续注意

- Windows 脚本中避免使用 `$home`、`$profile` 等容易和内置变量冲突的名字。
- 需要稳定写中文文档时使用 UTF-8 编码显式写入。

## 22. PowerShell 管道写中文文档会变成问号

### 现象

- 在当前 Windows 环境中，把大量中文文本通过 `PowerShell -> python stdin` 管道写入 Markdown 文件时，文件内容会被写成 `?`。
- 表面上写入成功，但再次读取字符码位时会发现中文已经丢失。

### 原因判断

- 问题不一定出在文件编码本身，而是出在 PowerShell 到下游进程标准输入的字符链路。
- 当前环境里，终端 / 管道对中文字符支持不稳定，尤其在长 here-string 加管道给 Python 时更容易触发。

### 已处理

- 本轮已改用 `Set-Content -Encoding utf8` 直接写中文文档，恢复并更新了规划文档和交接文档。
- 后续判断中文文档是否真的损坏时，不只看 PowerShell 输出，而是看 UTF-8 读取后的字符码位。

### 后续注意

- 需要写中文 Markdown 时，优先使用：
  - `Set-Content -Encoding utf8`
  - `.NET` 的 UTF-8 显式写入
  - 字节级写入
- 不要在当前环境里把整段中文规划文档通过 `@'...'@ | python -` 这种方式直接写回文件。
