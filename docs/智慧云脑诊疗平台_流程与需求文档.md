# 智慧云脑诊疗平台——完整流程与需求规范文档

## 1. 项目概述与设计理念

### 1.1 项目背景
在传统医院门诊管理系统（HIS）中，普遍存在流程繁琐、人工重复劳动多、纸质病历管理困难等痛点。传统的门诊系统虽然初步实现了信息化，但依然极度依赖窗口挂号人员、分诊护士、临床医生的大量机械性手工录入。
本项目旨在传统门诊HIS系统的业务底座上，进行深度“AI赋能”与架构升级，打造**智慧云脑诊疗平台**。该平台通过大语言模型（LLM）与深度学习影像识别技术，重构门诊全链路业务，实现高自动化、智能化的现代医疗服务流程。

### 1.2 核心设计理念（AI赋能与“无人医院”）
* **“无人医院”服务愿景：** 假设医院在无人工接待、无传统导诊台、甚至无传统窗口人员的前提下，患者依然可以通过手机端或智能自助端，全自主完成“主诉表达 -> 智能分诊 -> 方案推荐 -> 挂号支付”的闭环体验。
* **人机协同与人类最终确认原则（硬性指标）：** 尽管大模型在病情主诉生成、检查检验推荐、处方开立等环节具备强大的辅助决策能力，但系统必须严格贯彻**人类最高权威性**。凡涉及医疗安全的危险操作与核心诊断，系统必须设计“增/删/改”及“一键确认”机制，由执业医生进行人机协同决策，AI结果必须经过医生确认方能生效。
* **业务合规与创新性：** 项目严格遵循标准门诊流程大纲，同时鼓励各小组在基准要求下挖掘独特的细分需求，设计差异化的AI交互模式（如分诊算法创新、排班自然语言解析等），严禁出现完全相同的作品界面或业务逻辑。

## 2. 系统角色与用户端侧

### 2.1 患者端（移动互联端侧）
* **形态建议：** 纯手机端（H5/小程序/移动App）。
* **核心职责：** 自主注册登录、AI智能问诊、接收Top-3/Top-5医生推荐、线上挂号与模拟支付、查看历史费用记录与电子病历。

### 2.2 医生端（人机协作双屏模式）
* **形态建议：** 电脑端网页。
* **界面布局规范：** 建议采用“左传统、右AI”的双屏并行架构。
    * **左侧传统操作区：** 医生通过键盘、鼠标进行常规的病历查看、叫号、修改与确认等干活区域。
    * **右侧大模型绘画/对话辅助区：** 实时展示大模型提取的病情摘要、智能推荐的检查/处方清单、多模态医学影像分析结果。
* **核心职责：** 患者查看与叫号、病历首页确认、检查/检验申请开立、看诊记录更新、多模态影像结果查看、门诊确诊、电子处方开立。

### 2.3 管理员端（评估与统计侧）
* **核心职责：** 医生工作量综合统计、科室排班规则管理、大模型代码与软硬件能力评估、医生的诊疗水平能力技术评估（利用LLM对病历质量进行代码化或规范化评估）。

## 3. 核心业务流程规范

### 3.1 门诊主线业务流全链路
```
[患者端: 智能问诊] -> [LLM分诊与Top医生推荐] -> [模拟支付/挂号] 
       -> [医生端: 叫号接诊] -> [LLM自动生成病历初稿] 
       -> [AI辅助开立检查单(模糊态)] -> [患者缴费] -> [医技端: 影像摄取] 
       -> [深度学习模型: 影像智能识别(脑肿瘤)] -> [医生端: 查看影像结果与确诊] 
       -> [AI处方推荐] -> [医生确认/调整处方] -> [发药员: 药房发药]
```

1.  **AI智能问诊与分诊：** 患者登录手机端，向大模型口述或输入症状（如“头痛、持续三天、不发烧不咳嗽”）。大模型启动AI智能问诊，通过多轮对话挖掘症状细节，并引导患者完成分诊，自动判断推荐科室（如“神经外科”）。
2.  **智能医生推荐与挂号支付：** 系统摒弃传统HIS只看医生是否在岗的逻辑。AI分诊台根据患者的综合病症、出诊排班、医生的擅长领域（如某医生擅长脑肿瘤）、当前排队负载情况及病情紧急度，智能推荐Top-3或Top-5的心仪医生（支持性别偏好筛选）。患者选中医生后，调用微信/支付宝模拟接口完成支付（如每次支付1分钱）。
3.  **医生接诊与病历初填：** 门诊医生在工作站点击“叫号”，患者挂号信息自动推至医生端。大模型根据前期的“患者智能问诊对话记录”，自动提炼出主诉、现病史、既往史等信息，并直接回填至病历首页初稿中，极大地减少医生机械敲击键盘的时间。
4.  **AI辅助检查/检验开立（模糊态处理）：** 医生问诊过程中，面对复杂的“模糊态”病情，大模型会开启辅助提示（如提示“头痛除普通肠炎外，需警惕颅内肿瘤风险”），智能推荐一套合理的检查/检验组合清单。医生拥有最高权威，可对清单进行一键增删、调整，随后提交申请，患者完成检查项目的二次缴费。
5.  **多模态医学影像智能识别（核心算法项）：** 患者至检查科室进行影像摄取。本项目聚焦于**脑科肿瘤（医学影像图片）**。系统必须调用后台经过调优的机器学习/深度学习影像识别模型，对脑部CT/MRI图像进行灰度、边缘等特征提取，自动识别肿瘤位置、大小并给出辅助概率。该算法处理结果需打通并实时反馈给系统。
6.  **AI处方生成与电子病历确诊：** 门诊医生在“看诊记录”中调取已出影像结果的患者，右侧AI区展示肿瘤识别报告。医生完成“门诊确诊”后，进入“开设处方”界面。大模型根据确诊疾病自动拉出一堆推荐药品的清单（包含用法、用量、频次）。医生核对无误后点击“确认开立”，处方传送至药房，发药员进行“药房发药”完成闭环。

### 3.2 进阶AI挑战模块（采分加分项）
* **AI智能排班系统：** 颠覆传统在表格里逐一勾选时间段的复杂排班规则（传统的周一上午、周三下午等复杂矩阵）。利用大模型天然的自然语言解析能力，医生或管理员只需输入一句文本（如：“系统，我这周要去开会学习，另外周三下午有一台24小时的大手术，帮我调整下班表”），大模型即可自动解析逻辑并生成/剔除后台具体的排班记录。
* **AI分诊台调度：** 针对医院多个CT室/检查室，传统采用随机或轮流派单。AI分诊台能够根据各检查室的实时负载、排队时长、检查医生的擅长领域，以及患者的紧急度（如急诊绿色通道强行插队逻辑），交给大模型进行高维度的多因子智能动态调度。

## 4. 前后端模块与功能分解

### 4.1 前端模块及页面（Vue 3 + Element Plus）
* **财务挂号收费模块 (registration):**
    * `onsite_registration`: 窗口挂号/AI引导挂号页面（自动生成病历号、患者基本信息录入、结算类别选择）。
    * `registration_record`: 窗口退号与挂号状态追踪。
    * `expense_charge`: 门诊收费结算（多项待缴费检查/药品合并计算）。
    * `expense_refund`: 门诊退费处理。
    * `expense_manage`: 历史费用记录查询。
* **门诊医生模块 (physician):**
    * `physician_patient`: 挂号患者列表查看（排队人数、已看诊人数动态看板）。
    * `home_medical_record`: 病历首页（主诉、现病史、过敏史及LLM初填数据回显）。
    * `check_request` / `inspection_request`: 检查/检验申请单开立（含AI智能项目推荐组件）。
    * `physician_history`: 历史看诊记录调取。
    * `check_results` / `inspection_results`: 医学影像/检验数字化报告查看（集成脑肿瘤AI识别可视化区域）。
    * `outpatient_diagnosis`: 门诊确诊录入。
    * `write_prescription`: 开设处方页面（含AI处方推荐、自动合计金额、说明下拉列表）。
    * `disposal_request`: 处置申请（针对洗胃、清创等非药物品类）。
    * `expense_query`: 门诊医生侧费用综合查询。
* **检查/检验医生模块 (check & inspection):**
    * `check_apply` / `inspection_apply`: 医技申请单接收看板。
    * `check_patient` / `inspection_patient`: 患者录入、医技科室与医生二次分配。
    * `check_input` / `inspection_input`: 结果录入页面（支持医学影像图片上传与AI辅助识别算法触发）。
    * `check_manage` / `inspection_manage`: 历史医技工作量管理。
* **药房管理模块 (drugstore):**
    * `give_medicine`: 药房发药处理（根据患者缴费状态动态回显未发/已发列表）。
    * `refund_medicine`: 患者退药处理与库存回充。
    * `drug_storage`: 基础药库信息管理（药品编码、规格、警戒线提醒）。
    * `tran_history`: 药品交易流向记录。
* **处置医生模块 (disposal):**
    * `disposal_apply` / `patient` / `input` / `manage`: 门诊处置（非药、非影像类治疗）的排队叫号与结果录入。

### 4.2 后端微服务模块划分
系统强制采用**微服务架构**进行解耦，主要划分为以下核心业务组件：
1.  `his-common`: 通用支撑模块。封装统一的业务返回体（Result）、全局异常处理器、工具类（Util）、安全认证流。
2.  `his-outpatient` (端口: 8092): 门诊业务核心微服务。承载挂号、退号、收费、门诊医生站、医技检查检验处置的核心业务CRUD及逻辑。
3.  `his-drugstore` (端口: 8091): 药房药库微服务。专门承载发药、退药、药品维护、库存警戒线等高频并发交易。
4.  `his-ai-service`: AI与大模型专属微服务。集中封装FastAPI或Spring AI的大模型调用链、LangChain Agent调度、自然语言排班解析接口。
5.  `his-algorithm-service`: 多模态医学影像分析微服务。专门托管基于深度学习的脑部肿瘤影像特征提取与识别算法，提供高性能推理接口。

## 5. 技术栈与开发规范

### 5.1 前端技术栈（必须使用四大核心组件）
* **Vue 3：** 现代前端框架，全面采用 Composition API 进行高效的数据实时绑定与渲染。
* **Element Plus：** 基于 Vue 3 的组件库，保障全局页面组件（表格、对话框、表单）样式高度一致。
* **Pinia：** 官方推荐的现代化状态管理器，全面取代旧版 Vuex，负责全局登录用户（User Token）、部门类型（Dept Type）、患者上下文等核心状态维护。
* **Axios：** 异步请求框架，负责与后端微服务网关或独立端点进行全双工数据交互。

### 5.2 后端技术栈（双技术栈可选，严禁搞版本跨代）

#### 选项 A：Java 微服务技术栈
* **核心语言与版本：** JDK 17（硬性要求）。
* **核心框架：** Spring Boot 3.4 为底层核心（切勿使用 3.5+，以防 Spring Cloud Alibaba 组件不兼容）。
* **ORM 框架：** MyBatis / MyBatis-Plus。解开 SQL 与程序代码的高耦合，实现干净可测试的代码。
* **AI 接入层：** **Spring AI** 架构，用于标准化调用国内/外大模型接口。
* **微服务治理：** Spring Cloud Alibaba（Nacos 注册配置中心、Sentinel 限流熔断）。

#### 选项 B：Python 微服务/AI 技术栈
* **核心语言与版本：** Python 3.10 ~ 3.13 之间（切勿选 3.14/3.15 等过新版本，防止第三方算法包产生底层 Bug）。
* **核心 Web 框架：** **FastAPI**（利用其天然的异步协程与高性能，配合 Pydantic 进行出入参校验）。
* **AI 编排框架：** **LangChain 1.2** 为基准底座，支持引入 LangGraph、Deep Agent、Skill Agent 进行高级多 Agent 协同。

### 5.3 数据库与版本控制
* **关系型数据库：** **PostgreSQL (PGSQL)**（强烈建议将传统 HIS 的 MySQL 升级为 PGSQL。由于本项目涉及大量大模型交互，PGSQL 的 `pgvector` 插件能够对医学文本向量、AI 记忆、Embedding 向量数据提供极为出色的原生支持）。
* **项目构建工具：** Maven 3+（Java） / pip 或 conda（Python）。
* **版本控制系统：** **Git**（建立分布式团队团队库，强调个体离线工作与高频 Git 提交，降低多端协作复杂度）。

## 6. 数据库表结构设计（AI增强版）

### 6.1 数据表整体设计清单
1. `employee` (医院员工表): 存储医生、发药员等基本信息，外键关联科室与排班。
2. `department` (科室表): 医院科室划分（如神经外科、检验科）。
3. `regist_level` (挂号级别表): 阶梯号别与挂号限额管控。
4. `settle_category` (结算类别表): 存放患者的结算类别（自费、医保等）。
5. `scheduling` (排班表): 复杂星期排班规则。
6. `register` (患者历次挂号信息表): 记录挂号流水、模拟收费方式及当前的看诊状态。
7. `check_request` (检查申请表): 增强版，引入 AI 影像路径与 AI 判定字段。
8. `inspection_request` (检验申请表): 存储数字化检验指标。
9. `disposal_request` (处置申请表): 非药物性临床处置。
10. `medical_technology` (医技项目表): 检查、检验、处置基础字典表。
11. `medical_record` (患者病历表): 核心表，存储大模型初填字段与医生确诊字段。
12. `medical_record_disease` (病历首页疾病关联表): 病历与国际 ICD 编码疾病多对多关联。
13. `disease` (疾病表): ICD 国际疾病字典。
14. `prescription` (处方表): 存储开立的处方及 AI 处方标志。
15. `drug_info` (药品信息表): 药品规格、单价字典。
