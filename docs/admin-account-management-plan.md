## 本地演示流程（M1 + M2）

演示目标：确认管理员不是“页面演示登录”，而是能拿到 JWT、访问受保护的账号接口；同时确认医生、患者列表接口已经具备分页返回结构。

### 演示前提

1. 新环境先执行 `backend/migrations/20260714_02_create_admin_account_auth.sql`，在 `backend/.env` 配置 JWT 与管理员初始化变量后，进入 `backend` 运行 `python scripts/bootstrap_admin.py`。本机环境已完成这一步。
2. 启动后端服务。Windows 本地运行时，为避免控制台编码问题，可在 `backend` 目录执行：

   ```powershell
   $env:PYTHONUTF8='1'
   $env:PYTHONIOENCODING='utf-8'
   & 'D:\develop\Anaconda\envs\py3106\python.exe' run_microservices.py
   ```

3. 启动前端开发服务后，打开 `http://localhost:5173/admin/login`。账号和密码只从本机忽略提交的 `backend/.env` 获取；不要把密码或 JWT 粘贴到文档、截图或录屏中。

### 浏览器演示

1. 在管理员登录页使用已初始化的管理员账号登录，登录成功后应进入管理员工作台。
2. 打开账号管理页 `/admin/accounts`，确认医生、患者两个标签页都能加载首屏数据。
3. 本轮 M2 完成的是后端分页协议，当前页面仍只加载第一页；页码、上一页/下一页、每页条数控件属于 M3，演示时应明确说明尚未接入。

### 接口演示

1. 调用 `POST /api/v1/auth/admin/login`，请求体使用本地管理员凭据。成功响应会返回 `access_token`，仅在当前终端或 API 工具中临时使用。
2. 带上 `Authorization: Bearer <access_token>` 调用：

   ```text
   GET /api/v1/auth/admin/doctors?limit=1&offset=0
   ```

   响应应包含 `data.items` 和 `data.pagination`；其中 `pagination` 至少包含 `total`、`limit`、`offset`。
3. 修改 `offset`（例如改为 `1`），或使用 `limit=20&offset=20`，即可演示请求页发生变化。若本地数据总数不超过单页条数，后续页为空属于正常结果。
4. 不携带 Bearer Token 调用 `GET /api/v1/patient/admin/patients?limit=1`，应返回 `401`，用来证明患者档案管理接口已受管理员身份保护。

### 推荐讲解顺序

先展示未登录访问被拒绝（401），再展示管理员登录后访问列表成功（200），最后展示分页响应中的 `items + pagination` 结构，并说明前端翻页控件将在 M3 补齐。

2026-07-14 本地验证记录：管理员登录返回 200；带 Token 的医生列表返回 200 且包含分页结构；未带 Token 的患者管理列表返回 401；网关、认证、患者服务健康检查均正常。

## Goal

把 `/admin/accounts` 从“医生与患者基础资料的前 20 条卡片列表”升级为可在真实管理员身份下安全使用的人员与患者档案工作台：可检索、可分页、可区分加载/空/错误状态，并对敏感信息和关键操作保留最小必要的权限与审计边界。

## Current behavior

- 页面以 `doctor` / `patient` 两个 Tab 展示数据；两侧请求都固定为 `limit=20`，无 `offset`、`total` 或翻页控件。
- 医生端点 `GET /api/v1/auth/admin/doctors` 与患者端点 `GET /api/v1/patient/admin/patients` 都只返回数组；服务层仅执行 `LIMIT` 查询。
- 搜索失败时前端直接清空数组，页面会错误显示为“没有符合条件的账号”。
- 患者列表直接展示完整身份证号和家庭住址。
- 管理员登录只是浏览器 `sessionStorage` 中的演示身份；账号查询与修改接口未在服务端校验管理员身份。新增医生还会使用固定初始密码 `123456`。
- `Employee` 是带密码散列的医生/员工实体；`Patient` 是病历档案实体，并不存在患者账号凭据。因此当前页面名称“账号管理”混合了两个不同的业务概念。

## Scope and non-goals

本切片包含：

- 管理员登录的服务端身份与账号管理接口授权。
- 医生和患者列表的服务端 offset 分页与前端翻页。
- 账号页错误状态、敏感数据脱敏、列表信息层级和可访问性。
- 医生账号的新增初始凭据与重置密码边界；医生停用/启用。
- 对账号管理写操作记录操作者、目标和操作类型。

本切片不包含：

- 把 `Patient` 改造成可登录的认证账号，或把患者与员工强行合并到一张用户表。
- RBAC 扩展到排班、药房、账单等所有后台模块；本次先建立可复用认证/授权基础，并优先覆盖账号模块的读写接口。
- 物理删除医生、患者或任何已有医疗记录主体。
- 修改既有医生端登录的业务流程，除非管理员凭据模型复用时发现不可避免的冲突。

## Proposed solution

### 1. 先完成认证架构决策（implementation gate）

在写业务代码前确认以下 ADR，并记录到本文档或单独 ADR：

1. 管理员使用独立 `AdminAccount` 身份，不复用 `Employee`，因为管理员与医生职责、最小权限和生命周期不同。
2. 登录成功后由后端签发短期、可验证的管理员 access token；前端只保存最小身份展示信息和令牌，不再把任意工号写入 `sessionStorage` 即视为登录。
3. 后端依赖 `require_admin` 统一验证令牌、身份状态和权限；前端路由守卫仅改善体验，绝不作为安全边界。
4. 管理员帐号初始化、令牌签名密钥和首个管理员创建方式必须由环境变量/部署脚本提供，不能把默认管理员密码或令牌写入代码、seed 或前端。

若现有依赖不具备安全的令牌签发/验证能力，先选择一个与 FastAPI 兼容、可维护的方案并补依赖锁定；不得以沿用 `X-Admin-Token` 或浏览器常量替代用户登录。

### 2. 明确数据与页面语义

- 页面名称调整为“人员与患者档案管理”；Tab 保持“医生”“患者档案”。
- 医生 Tab 管理 `Employee` 的工作资料和账号状态；患者 Tab 管理 `Patient` 的就诊档案。页面文字不再暗示患者拥有可登录密码。
- 病例号保留为患者列表主标识；身份证号默认仅显示掩码（如 `2101**********1234`），地址默认不在列表展示。查看完整信息应是受权限控制的显式操作，并留下审计记录。
- 医生删除改为可恢复的启用/停用；停用前校验未来排班、待接诊/未完成流程，并返回可读的阻断原因。患者档案不提供删除或停用操作。

### 3. 分页接口契约

将两类列表改为同一分页形状，并保持关键词过滤、排序和总数使用同一组条件：

```json
{
  "items": ["..."],
  "pagination": {
    "total": 125,
    "limit": 20,
    "offset": 40
  }
}
```

- 请求参数：`keyword?: string`、`limit?: int`、`offset?: int`；服务端统一钳制为 `1 <= limit <= 100`、`offset >= 0`。
- 医生按 `Employee.id DESC` 稳定排序；患者改为 `Patient.created_at DESC, Patient.id DESC`，避免同一创建时间导致跨页重复或遗漏。
- 分别执行分页查询和相同筛选条件下的 `COUNT(*)`，不要通过当前页长度猜测总数。
- API 类型、接口响应和服务层实现同步更新；不得为保留旧数组返回而加入前端猜测分支。
- 参考并复用已有账单/药房/审计的 `items + pagination` 约定，不新增另一种页码协议。

### 4. 前端交互与状态

- 医生和患者分别维护 `keyword`、`limit`、`offset`、`total` 和加载状态，切换 Tab 不丢失另一侧的检索位置。
- 提交搜索、切换每页条数、切换 Tab 的初次加载均从第一页开始；新增、编辑、启用/停用、重置密码后刷新当前列表。若当前页因数据变化为空且不是第一页，回退一页再加载。
- 列表头显示“显示 X–Y / 共 Z 条”、每页条数（20/50/100）和上一页/下一页；无总数、正在加载和无权限时不显示误导性页码。
- loading 使用现有后台视觉语言的骨架/禁用状态；空状态说明是“未匹配检索条件”还是“尚无记录”；请求失败展示可重试错误区，不清空已成功加载的数据。
- 所有按钮补齐 hover、focus-visible、disabled、loading 状态；窄屏下工具栏和分页控件纵向排列，卡片不横向溢出。沿用 `admin-skin.css` 的颜色 token、圆角与按钮样式，不新增页面孤岛设计系统。

### 5. 凭据、权限与审计

- 新增医生不再由前端传递或后端默认 `123456`。改为管理员选择“发送/生成一次性初始凭据”或创建后执行显式“重置密码”；原始密码只在一次性受控交付流程出现，后端不记录明文。
- 新增 `must_change_password`（或等价状态）和密码重置的有效期/撤销语义；首次使用必须修改初始凭据。
- 写接口（新增、编辑、启用/停用、重置密码）要求明确权限；读接口至少要求已认证管理员。
- 记录操作者管理员 ID、目标实体 UUID、操作类型、结果、时间和必要的失败原因；日志不保存密码、完整身份证号或家庭住址。
- 管理员 token 过期、失效或账号停用时，API 返回一致的 401/403；前端清除本地会话并回到登录页，保留提示而非静默跳转。

## Implementation steps

1. **认证基础与迁移**
   - 勘查配置、部署和数据库迁移机制；落地 ADR。
   - 新增管理员身份、密码散列、账号状态、令牌签发/验证和 `require_admin` 依赖。
   - 用受控配置初始化管理员，不提交任何凭据。

2. **收紧账号模块接口**
   - 为医生、患者的列表和写接口接入管理员依赖。
   - 实现医生/患者分页查询与响应模型；更新 `auth.ts`、`admin.ts` 类型。
   - 移除固定默认医生密码，增加显式重置/初始凭据接口和医生启停接口。
   - 写入账号操作审计，并确保序列化与日志均排除敏感字段。

3. **改造账号管理页面**
   - 将 `AdminLoginView` 改为真实登录调用；调整 `adminSession` 和 HTTP 请求拦截器的令牌处理。
   - 改造 `AdminAccountsView` 的分页状态、搜索重置、重试、错误/空/加载状态和窄屏布局。
   - 将患者敏感字段脱敏；为完整资料增加权限明确的详情操作；加入医生启停和重置凭据的二次确认。

4. **回归与文档同步**
   - 同步 `docs/frontend-plan.md`、API 文档和管理员端功能规划中的认证边界与页面名称。
   - 在非生产数据环境完成登录、分页、脱敏、写操作审计和令牌失效回归。

## Risks

- 管理员真实鉴权涉及新的身份和部署密钥，是本任务中风险最高的变更；必须先完成 ADR 和配置方案，不能把演示账号“加一层前端校验”当作修复。
- 现有管理员页面和接口普遍处于演示/无授权状态。仅保护账号页能降低本模块风险，但不等于整个后台已安全；其余模块需要后续统一迁移。
- `Patient` 记录承载医疗数据。脱敏、详情授权和审计必须在接口层实现，不能只依赖前端隐藏。
- offset 分页在数据持续写入时可能出现轻微页漂移；本期采用稳定排序降低风险。若将来出现高频写入和长列表浏览，再评估 cursor pagination。
- 医生停用会影响排班、挂号和接诊链路，必须先明确阻断规则并做跨服务回归。

## Validation strategy

### Backend

- 单元/服务测试：医生和患者的默认页、关键词、`offset`、边界参数钳制、稳定排序与 `total`。
- API 契约测试：未登录 401、无权限/停用账号 403、正确管理员访问 200；确保响应中不存在密码散列、完整敏感字段或令牌。
- 写操作测试：创建医生不接受隐式默认密码；重置凭据、首次改密标志、启用/停用阻断规则和审计记录。
- 运行既有 `test_auth_admin_accounts.py`、`test_patient_admin_accounts.py`，并新增端点/鉴权测试，不以 mock 绕过授权依赖。

### Frontend

- `npm run build` 与 TypeScript 检查。
- 组件或浏览器回归：登录成功/失败/令牌过期；两 Tab 搜索、翻页、每页条数、创建/编辑后的页码修正；空态、错误重试、加载禁用。
- 1280px、980px、760px 宽度下检查分页工具栏、卡片和对话框，确认无横向溢出且键盘焦点可见。

### Integrated acceptance

1. 未认证请求账号列表和写接口被拒绝。
2. 管理员登录后，可检索并翻至第 2 页；显示总数与范围正确。
3. 患者列表默认不暴露完整身份证号和地址。
4. 新增或重置医生凭据不会在前端、接口响应或审计日志中泄露明文。
5. 启停、编辑和重置均记录可追溯操作者，且不破坏已有排班/接诊数据。
