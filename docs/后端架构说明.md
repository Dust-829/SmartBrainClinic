# 后端架构说明

本文档记录当前后端实现，不描述早期 Java/Spring 规划。

## 架构概览

当前后端采用 FastAPI 微服务架构。每个业务域独立 FastAPI 应用，统一由 Gateway 暴露入口。服务间通信同时使用 HTTP 内部客户端、RabbitMQ 异步事件和 Nacos 服务发现 fallback。

```text
Client / Frontend
       |
       v
Gateway :8000
       |
       +-- Auth :8001
       +-- Patient :8002
       +-- Medical :8003
       +-- Pharmacy :8004
       +-- Billing :8005

Infrastructure:
  PostgreSQL + pgvector
  RabbitMQ
  Redis
  Nacos
```

## 服务职责

### Gateway

入口文件：`backend/app/main.py`

- 接收 `/api/v1/{service}` 请求。
- 根据 `service` 选择目标微服务。
- 优先从 Nacos 获取服务实例。
- Nacos 不可用时使用环境变量或本地 fallback URL。

### Auth Service

目录：`backend/app/microservices/auth`

- 科室、诊室、员工、医生、挂号级别、结算类别。
- 医生擅长向量生成与相似医生检索。
- 医生 AI 评分调整。
- 目前更接近“基础资料与员工服务”，完整认证授权仍需补齐。

### Patient Service

目录：`backend/app/microservices/patient`

- 患者建档。
- 挂号、退号、线上预挂号。
- 排班规则、实际排班、号源时段。
- AI 分诊、医生推荐。
- 候诊队列 WebSocket。
- 排班异常工单与自动处理。

### Medical Service

目录：`backend/app/microservices/medical`

- 病历、诊断、疾病字典。
- 检查、检验、处置申请。
- AI 病历草稿。
- 医学影像推理接口。
- 接收挂号和缴费消息，推动医疗状态流转。

### Pharmacy Service

目录：`backend/app/microservices/pharmacy`

- 药品资料与库存。
- 处方和处方明细。
- 发药与退药。
- AI 处方推荐。
- 接收缴费和退费消息，推动处方状态流转。

### Billing Service

目录：`backend/app/microservices/billing`

- 合并缴费。
- 账单和账单明细。
- 退费。
- 通过 Outbox 发布支付和退款事件。

注意：当前路由前缀为 `/api/v1/bill`，部分配置和说明中可能仍使用 `billing`，后续需要统一。

## 公共模块

目录：`backend/app/common`

- `config.py`：公共配置基类。
- `database.py`：异步数据库引擎和 session 工厂。
- `mq.py`：RabbitMQ 客户端。
- `nacos_client.py`：Nacos 注册与服务发现。
- `clients.py`：服务间 HTTP 客户端基础能力。
- `ai_embedding.py`：向量嵌入调用。
- `response.py`：统一响应封装。
- `enums.py`：业务状态枚举。

## 数据与消息

### 数据库

- 当前使用 PostgreSQL。
- 各服务当前共享 `his_db`。
- ORM 使用 SQLModel + SQLAlchemy Async。
- 向量字段使用 pgvector。

### 消息队列

- RabbitMQ 用于跨服务异步事件。
- 多个服务使用 Outbox 表记录待发布事件。
- 后台 sweeper 扫描 pending 事件并发布到队列。
- 消费者处理支付、退款、挂号等状态同步。

## 已知工程风险

- 真实 LLM API Key 不应硬编码在 `config.py`。
- Docker 环境下 Nacos 注册 `127.0.0.1` 会导致服务发现地址错误。
- `bill` / `billing` 命名需要统一。
- 缺少自动化测试和 CI。
- 缺少完整鉴权、权限控制和审计日志。
- 前端尚未初始化。

## 建议演进顺序

1. 配置安全与 `.env.example`。
2. 服务命名统一。
3. Docker/Nacos 注册地址修正。
4. 最小业务闭环 API 验证。
5. 自动化测试。
6. 前端初始化。
