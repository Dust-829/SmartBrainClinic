# AI 检查检验建议实施计划

## Goal

在医生接诊页提供 AI 辅助的检查、检验候选项目。医生可以查看理由、选择或删除候选项；检查项目还必须确认或修改检查部位和目的。只有医生把项目加入现有“待签医嘱”清单并点击统一签署，系统才会创建真实订单。

## Current Behavior

- 接诊页已加载真实的 `check`、`inspection`、`disposal` 医技目录，并通过 `POST /api/v1/medical/orders/sign` 统一签署。
- 已确认病历保存主诉、现病史、既往史、查体、诊断和过敏史；患者服务已提供挂号 AI 上下文读取能力。
- 已开检查、检验项目可按挂号读取；候选推荐接口只在病历确认后按需生成，不创建订单。
- AI 处方建议与正式开方分离，检查检验建议应保持同一安全边界。

## Proposed Solution

### 1. 只读建议接口

新增 `POST /api/v1/medical/orders/ai-recommendation`，请求只包含 `register_uuid`。接口必须：

- 要求该挂号存在且病历已经由医生确认。
- 读取病历、可用的真实医技目录、已开检查和已开检验项目；患者 AI 分诊上下文读取失败时降级，但不能阻塞建议。
- 仅返回目录中仍可用且尚未开立的 `check`、`inspection` 项目；不返回 `disposal`，不创建订单，不修改病历。
- 对每一项返回 `type`、`medical_technology_id`、`tech_code`、`tech_name`、`price`、`reason`。`check` 还返回建议的 `check_position` 与 `check_info`，供医生修改。
- 返回来源与提示，例如 `record_catalog_rule`、`llm_catalog_validated`、`fallback`；当信息不足或无法安全匹配时返回空候选，不以猜测项目替代。

### 2. AI 与安全校验

- 先由服务端建立候选白名单，再允许 LLM 在白名单 ID 内排序并说明理由；LLM 不能生成目录外项目、价格或订单。
- LLM 调用失败、返回 JSON 不合法、项目类型不匹配、项目已开立或建议重复时，丢弃该结果并使用受控规则候选或空列表。
- 不把建议当作诊断、医嘱或紧急处置结论。页面必须继续显示“仅供医生参考，医生确认后才会开立”。
- 第一版不增加业务数据表：建议是临时读取结果，不保存为检查单或检验单。若调用真实 LLM，沿用现有 AI 审计能力记录调用元信息，严禁记录密钥。

### 3. 前端接入

在右侧 AI 支持区增加“AI 检查检验建议”卡片，仅在病历确认后允许点击生成。

- 候选项默认不进入待签清单。
- 医生可选择、删除或编辑；点击“加入待签清单”后复用既有 `pendingOrders`。
- `check` 候选进入清单前必须有非空的检查部位和目的，继续由现有签署校验兜底。
- 前端不直接调用创建检查或检验接口，也不绕过 `POST /api/v1/medical/orders/sign`。

## Delivery Slices

1. 定义接口合同、只读数据边界和测试用例。本文件完成后进入后端接口实现。
2. 实现医疗服务只读建议接口：确认病历、目录过滤、已开项目去重、受控规则候选和单元测试。
3. 接入可选 LLM 排序与理由生成，并对输出做目录白名单和结构校验。
4. 在接诊页展示候选项，支持选择、删除、编辑并加入现有待签清单。
5. 增加浏览器回归：确认病历后生成建议，检查医生编辑后进入待签清单；断言生成建议本身不产生真实订单。

完成状态：五个实施切片均已完成。已完成受控规则候选核心、`POST /api/v1/medical/orders/ai-recommendation` 只读接口，以及接诊页右侧候选卡。接口要求已确认病历，读取实时目录与已开项目并在分诊上下文缺失时降级。医生可忽略候选；对检查候选可修改检查部位和目的后，明确加入既有待签清单。此动作只修改浏览器内存，不产生医疗订单或病历写入；仍须通过既有 `POST /api/v1/medical/orders/sign` 统一签署，才会创建真实订单。浏览器回归覆盖了“加入待签清单后订单数不变”。

LLM 排序：接口会在规则候选非空且已配置 `LLM_API_KEY` 时调用现有 `AIClient`，模型只可返回候选 ID 的排序和理由。类型、价格、检查部位和检查目的始终使用服务端白名单中的原始数据。目录外、重复、缺理由或格式异常的结果会被丢弃；没有有效结果、模型未配置或调用失败时回退到规则候选。单次模型调用最多等待 8 秒且不重试，避免阻塞医生页面。每次尝试真实 LLM 都会写入脱敏 AI 审计，审计失败不会影响建议接口；该审计不是医疗订单或病历写入。

真实验证：2026-07-15 已重启 Medical 服务并通过受控 Playwright 主链。最新一次模型调用在约 8.1 秒内返回无效载荷，接口正确以 `rule-order-catalog` 降级，审计记录包含 `llm_order_invalid_payload_fallback` 和 `llm_order_no_valid_result_fallback`，状态为待审；整个回归没有在候选加入待签清单时创建检查、检验或处置订单。

## Risks

- 检查项目缺少部位或目的时不能签署，不能用默认值绕过该约束。
- 患者 AI 分诊上下文和 LLM 都可能不可用，接口必须可降级并清楚标识来源。
- 必须基于实时目录和已开项目计算，避免推荐已停售或重复收费项目。
- 本功能是辅助决策，不应扩大为自动诊断、自动开单或治疗方案。

## Validation Strategy

- 后端：覆盖未确认病历拒绝、已开项目排除、目录外项目拒绝、LLM 失败降级和无写入副作用。
- 前端：`pnpm run build`；候选项不自动进入待签清单，检查项缺部位或目的时不可签署。
- 浏览器：从确认病历到加入待签清单的完整路径；生成建议后回查该挂号没有新增检查或检验订单。
