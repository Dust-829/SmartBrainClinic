# 患者端就诊码概念展示计划

> 状态：待实施。此功能仅用于患者端产品演示，不生成真实凭证、不接入扫码设备或院内核验流程，也不等同于国家医保码。

## Goal

让评审者在患者端直观看到“出示就诊码”的典型就医场景：登录后的患者可打开一张包含二维码、门诊号和就诊提示的展示卡片。

## Current behavior

- 首页与个人中心已有“就诊码”入口。
- 个人中心当前错误地复用了模拟支付的 `qr_code_url`；该字段不是可展示的二维码图片，页面也无法稳定呈现概念效果。
- 患者端已有姓名和门诊号等登录后资料，足够支撑展示卡片，无需新增后端数据表或接口。

## Proposed solution

1. 新增受保护路由 `/patient/visit-code`，首页和个人中心的“就诊码”入口统一跳转到该页。
2. 新增 `PatientVisitCodeView.vue`，复用当前患者端的蓝色移动端视觉语言，展示：
   - 固定的前端二维码图形（SVG/CSS 生成）；
   - 患者姓名、门诊号和“本人”标记；
   - “请向导诊台或诊室工作人员出示”的说明；
   - 明确的“演示用就诊码，不支持实际扫码核验”提示。
3. 二维码内容使用固定的非敏感演示字符串；不放身份证号、患者 UUID、挂号 UUID、支付 URL 或任何可识别数据。
4. 不新增 QR library、后端 API、数据库迁移、staff 核验页、摄像头权限或倒计时刷新逻辑，保持改动只在患者端展示层。
5. 移除个人中心中把 `qr_code_url` 当图片渲染的旧弹窗，避免把模拟支付与就诊码混淆。

## Delivery slices

1. 在 router 中添加 `/patient/visit-code` 并沿用现有患者登录守卫。
2. 实现概念展示页和静态二维码图形；处理未登录跳转、窄屏排版和页面返回。
3. 修改首页、个人中心的入口指向，并删除支付 URL 复用逻辑。
4. 同步 `docs/frontend-plan.md`、运行前端构建和浏览器视觉回归。

## Risks

- 该页面是视觉/流程概念，不能被描述为真实身份核验能力。
- 固定二维码可以被截图；因为它不参与任何业务操作，这是可接受的展示限制，但页面必须明确标注。
- 后续若改为真实院内核验，应重新评审鉴权、短时 token、防重放、审计与 staff 授权，不能在本计划上直接叠加。

## Validation strategy

- `npm run build` 通过。
- 登录后可从首页、个人中心进入；未登录时遵循现有路由守卫跳转登录页。
- 360--430px 宽度下二维码、姓名、门诊号和说明完整可读，无横向溢出。
- 检查页面和代码中不再将 `qr_code_url` 作为就诊码图片，也不展示任何敏感标识。
