# 实训需求与项目实施计划

本文档将实训要求、当前项目真实进度、差距分析和后续实施计划合并到一处，作为后续开发、答辩和 PPT 汇报的统一依据。

## 1. 项目定位

项目名称为“智慧云脑诊疗平台”，定位是 AI 赋能的门诊 HIS / 无人 AI 医院实训项目。项目只覆盖门诊诊疗全链路，不涉及住院业务。

当前建议将业务主线固定为 **脑科 / 神经外科门诊**，以“头痛、恶心、视物模糊、疑似脑部占位”等场景为核心，重点展示 AI 问诊、AI 分诊、医生推荐、脑部 CT/MRI 影像识别、医生审核、电子病历和处方流转。

项目最终技术路线已经确定为：

- 后端：Python FastAPI 微服务。
- AI 编排：LangChain / LangGraph。
- 数据库：PostgreSQL + pgvector。
- 前端：Vue 3 + Pinia + Axios + Element Plus。
- 基础设施：RabbitMQ、Redis、Nacos、Docker Compose。

Java / Spring Boot / JDK17 / Maven 3.9+ 是实训允许路线和早期规划背景，但当前项目不再采用 Java 作为最终后端实现。

## 2. 总体业务流程

项目需要展示一条完整的门诊闭环：

```text
患者张三注册
  -> 向大模型描述症状
  -> AI 判断是否需要到院就诊
  -> AI 分诊推荐神经外科/脑科
  -> AI 从当班医生中推荐 3 到 5 位医生
  -> 张三选择医生、科室、号源
  -> 线上或线下挂号
  -> 支付并进入候诊队列
  -> 张三在时间段内报到
  -> 医生华佗叫号接诊
  -> 医生初步诊断
  -> AI 给出检查/检验建议
  -> 医生增删改检查/检验项目并确认
  -> 患者缴费
  -> 医技端执行检查和检验
  -> 检查产生脑部 CT/MRI 图片，检验产生数字报告
  -> AI 影像模型输出肿瘤概率、风险提示和辅助报告
  -> 医生审核 AI 诊断建议
  -> AI 根据诊断和结果推荐处方
  -> 医生审核并确认处方
  -> 财务缴费
  -> 药房发药
  -> 电子病历归档
```

挂号到医生接诊之间的业务细节可以简化，但系统必须体现挂号单关联患者、科室、医生、排班和号源时段，并支持线上/线下挂号、候诊队列、医生叫号和状态流转。

## 3. 角色端设计要求

至少需要三个端，并且每个角色界面必须明显不同。

| 角色端 | 推荐形态 | 必须体现的差异 |
| --- | --- | --- |
| 患者端 | 手机端优先 | 注册、AI 问诊、分诊结果、医生推荐、号源选择、挂号支付、候诊状态、历史记录 |
| 医生端 | 电脑端 | 左侧 HIS 人工业务区，右侧 AI 辅助区；用于接诊、病历、检查检验、影像、诊断、处方审核 |
| 管理员端 | 电脑端 | 排班规则、AI 排班审批、医生评价、统计看板、AI 能力评估 |
| 医技端 | 电脑端或工作台 | 检查/检验申请接收、影像上传、数字报告录入、AI 影像结果查看 |
| 药房端 | 电脑端或工作台 | 待发药、发药、退药、库存、补货建议 |
| 财务端 | 电脑端或工作台 | 挂号缴费、检查/药品缴费、退费、账单查询 |

答辩最低要保证患者端、医生端、管理员端；医技、药房、财务可以作为扩展端或工作台模块。

医生端建议重点做“双屏”：

- 左侧：传统 HIS 操作区，包括候诊队列、病历、检查检验、诊断、处方。
- 右侧：大模型辅助区，包括症状摘要、检查建议、影像 AI 报告、诊断建议、处方建议。

## 4. AI 与医生审核原则

项目需要体现“无人 AI 医院”的设想，但当前实现和答辩主线应以医疗安全为前提，采用 **AI 生成建议，医生审核确认后生效** 的模式。

| 环节 | AI 产生的信息 | 最终生效方式 |
| --- | --- | --- |
| AI 问诊 | 症状摘要、就医建议 | 患者确认后进入挂号 |
| AI 分诊 | 推荐科室 | 患者选择或确认 |
| 医生推荐 | 3 到 5 位候选医生 | 患者选择 |
| 病历草稿 | 主诉、现病史、既往史等 | 医生修改并确认 |
| 检查检验建议 | 推荐检查/检验清单 | 医生增删改并确认 |
| 医学影像识别 | 肿瘤概率、风险提示、影像报告 | 医生审核后进入诊断 |
| 诊断建议 | 疾病判断、风险说明 | 医生最终确认 |
| 处方建议 | 药品、用法、用量、频次 | 医生审核后开立 |
| 排班建议 | 自然语言排班转记录 | 医生或管理员审批 |

数据设计上必须把 AI 产生的信息和人工确认的信息分开。AI 草稿、AI 推荐、AI 影像结果不能直接等同于最终医疗结论；最终病历、诊断和处方要有医生确认标记。

## 5. 医学影像与检验要求

医学影像是本项目的关键考核点。

要求如下：

- 聚焦脑部 CT/MRI，服务神经外科/脑科场景。
- 检查产生图片，检验产生数字化报告。
- 医学影像处理必须体现机器学习/深度学习模型。
- 影像结果需要输出概率、报告、风险提示，后续最好补病灶位置或可视化标注。
- 医学影像标准需要体现 PACS / DICOM。
- AI 影像结果要打通医生端、电子病历、诊断和处方流程。

当前代码中 Medical Service 已支持检查结果录入、`image_path`、`ai_tumor_prob` 和 AI 影像报告，但 `ai_image_inference.py` 仍是 Mock 推理。后续如果要满足“深度学习调优后的模型”要求，需要接入真实模型服务，或者至少把模型服务边界、输入输出和数据集说明补齐。

检验报告当前模型支持 JSONB 数字结果，适合存血常规、生化指标等结构化报告，但还需要演示数据和前端展示。

## 7. 与实训要求的差距

当前最主要差距如下：

1. **前端缺失**：患者端、医生端、管理员端还没有界面，无法体现“每个角色界面不相同”。
2. **医生双屏未实现**：后端有病历、检查、影像、处方能力，但 UI 尚未体现左 HIS / 右 AI。
3. **医学影像仍是 Mock**：当前只是根据图片路径模拟结果，不是真实机器学习/深度学习模型。
4. **PACS/DICOM 未形成完整数据流**：需求里需要体现 DICOM 标准，当前代码层面还未接入。
5. **AI 建议和人工确认需要统一**：病历已有确认字段，但检查建议、处方建议、影像建议还需要统一“AI 草稿/医生确认”的数据规范。
6. **演示数据缺失**：张三、华佗、神经外科、脑部 CT/MRI、检验报告、药品、排班等需要固化。
7. **AI 分诊台调度 CT 室不完整**：已有医技人员推荐思路，但多 CT 室负载调度、急诊插队还没有完整闭环。
8. **AI 药房不足**：已有药品和库存，缺少补货建议、库存预测。
9. **权限和审计不足**：医疗系统必须有角色权限和关键操作审计。
10. **测试不足**：挂号抢号、缴费退费、库存扣减、消息投递等高风险流程缺少测试。
11. **配置安全风险**：LLM API Key 不应硬编码在代码中，需要移入 `.env` 并轮换已暴露密钥。

## 8. 数据设计补充要求

后续数据库和 PPT 说明中需要明确：

- 挂号信息必须关联患者、科室、医生、排班和号源时段。
- HIS 主体数据要素包括医生、患者、药品、检验检测设备。
- 结果主体包括检查影像、检验报告、诊断、处方、电子病历。
- AI 产生的信息和医生/人工产生的信息要分开保存。
- 处方和医嘱可以在业务上放在同一医疗执行链路中说明，但数据上建议区分：处方偏药品，医嘱偏检查、检验、处置等执行指令。
- 医学影像数据需要说明 PACS / DICOM 标准，以及系统如何保存影像路径、AI 结果和医生确认结果。

## 9. 后续实施计划

### 阶段一：统一文档和演示主线

目标：让项目方向统一，避免 Java 旧规划和 FastAPI 当前实现混在一起。

- 将需求文档中早期 Java/Spring 服务命名改为当前 FastAPI 微服务命名。
- 明确本项目最终后端路线是 Python FastAPI。
- 固定答辩剧本：患者张三、医生华佗、神经外科、脑部 CT/MRI。
- 补充 DICOM/PACS、AI 建议审核、处方/医嘱关系说明。
- 整理一份演示数据清单。

### 阶段二：补前端三端最小闭环

目标：满足“至少三个端，且界面不同”的要求。

- 患者端：移动端布局，完成注册、AI 问诊、医生推荐、挂号、支付、候诊状态。
- 医生端：桌面双屏布局，完成队列、病历确认、检查检验建议、影像结果、诊断、处方。
- 管理员端：完成排班、AI 排班审批、统计和基础资料管理。

### 阶段三：打通可演示业务闭环

目标：从张三注册到药房发药可以连续演示。

- 准备固定演示数据。
- 跑通患者建档、AI 分诊、医生推荐、挂号、支付、候诊。
- 跑通医生接诊、AI 病历草稿、检查检验开立、缴费。
- 跑通医技结果录入、影像 AI 结果、医生确认。
- 跑通 AI 处方推荐、医生确认、缴费、药房发药。

### 阶段四：强化医学影像

目标：把影像模块从 Mock 提升到可答辩的 AI 模型服务。

- 明确 DICOM/PACS 数据流。
- 用真实或半真实脑部 CT/MRI 示例替换纯 Mock。
- 将 `ai_image_inference.py` 替换或封装为模型服务调用。
- 输出概率、报告、医生确认状态，并在医生端可视化。

### 阶段五：补 AI 扩展能力

目标：覆盖加分项。

- AI 排班：自然语言规则转排班记录，管理员审批。
- AI 分诊台：按 CT 室负载、医生擅长和紧急程度调度。
- AI 药房：库存预警、补货建议、药品消耗趋势。
- AI 质量评估：病历质量、处方合理性、AI 建议采纳率。

### 阶段六：工程稳定性和安全

目标：让系统能稳定演示并具备基本工程质量。

- 移除硬编码密钥，补 `.env.example`。
- 修正 Docker/Nacos 注册地址。
- 统一 `bill` / `billing` 命名。
- 补角色登录、权限控制和操作审计。
- 补核心测试：挂号抢号、缴费退费、库存扣减、Outbox 消息。
- 增加一键启动和一键演示脚本。
