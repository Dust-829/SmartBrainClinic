# 当前项目回顾

更新时间：2026-07-01

本文档用于快速回顾 SmartBrainClinic 当前真实状态、已经完成的工作、遇到的问题、接下来怎么做，以及本次准备推送到 GitHub 的改动范围。更完整的总计划见 `docs/项目规划.md`，前端专项计划见 `docs/frontend-plan.md`。

## 1. 项目定位

SmartBrainClinic 是面向门诊流程的 AI 赋能 HIS / 无人 AI 医院实训平台。当前实现路线是 Python FastAPI 微服务后端 + Vue 3 前端，不再按早期 Java / Spring Boot 规划推进。

当前固定演示主线是脑科 / 神经外科门诊，围绕“头痛、恶心、视物模糊、疑似脑部占位”等症状，串联患者建档、AI 分诊、医生推荐、挂号、支付、候诊、医生接诊、检查检验、影像辅助分析、处方和电子病历归档。

## 2. 当前技术状态

后端：

- Gateway：`backend/app/main.py`，端口 `8000`。
- Auth：端口 `8001`。
- Patient：端口 `8002`。
- Medical：端口 `8003`。
- Pharmacy：端口 `8004`。
- Billing：端口 `8005`。
- 基础设施：PostgreSQL + pgvector、Redis、RabbitMQ、Nacos、Docker Compose。

前端：

- 路径：`frontend/`。
- 技术栈：Vue 3 + Vite + Vue Router + Pinia + Axios + Element Plus。
- 当前开发服务地址：`http://localhost:5173`。
- 当前重点：患者端移动端主链路。

项目级协作工具：

- 已新增 `PRODUCT.md`，用于给设计和 agent 提供产品上下文。
- 已新增 `.agents/AGENTS.md`，用于记录后续 agent 接手规则。
- 已安装 `.claude/skills/impeccable/`，用于前端设计和一致性检查。

## 3. 已完成的主要工作

### 3.1 后端与联调基础

- 网关已补 CORS，允许 `http://localhost:5173` 和 `http://127.0.0.1:5173` 访问，解决浏览器注册时的 CORS 型 `Network Error`。
- 后端已有患者建档、患者查询、AI 分诊、科室列表、医生推荐、排班、线上挂号、支付模拟、候诊状态、挂号历史等接口。
- 医生推荐依赖真实科室、真实医生、真实排班和真实号源，不用前端假医生补页面。

### 3.2 患者端页面

已经从占位首页拆成真实页面和路由：

- `/patient`：患者登录。
- `/patient/register`：注册建档。
- `/patient/home`：患者首页。
- `/patient/departments`：按科室挂号入口。
- `/patient/triage`：AI 问诊。
- `/patient/doctors`：医生推荐。
- `/patient/confirm-register`：确认挂号。
- `/patient/payment`：线上支付。
- `/patient/queue`：候诊状态。
- `/patient/registers`：挂号记录。
- `/patient/profile`：个人中心。

### 3.3 患者端交互与设计

- 患者端已转向蓝白医疗移动端风格，首页、挂号记录、个人中心和主链路页面逐步统一。
- 首页已经不再单独放“智能分诊”入口，智能分诊集成在按科室挂号流程中。
- 从首页进入按科室挂号时，自动弹出“是否需要智能分诊帮助”的提示；从 AI 问诊返回后不重复弹。
- 挂号页顶部提示条可以直接进入 AI 问诊。
- 挂号链路统一引入 `PatientFlowHeader.vue`：左侧返回业务上一级，右侧一键返回患者首页。
- 底部导航第二项已固定为“挂号记录”，进入 `/patient/registers`。
- 患者端不再展示 `SJWK` 等内部科室编码，展示层使用中文科室名。

### 3.4 登录注册

- `/patient` 是患者登录页，使用姓名 + 证件号查询真实患者档案。
- `/patient/register` 使用真实建档接口。
- 注册成功后直接进入已登录患者流程。
- 身份证输入会自动匹配出生日期，并校验身份证出生日期是否合法。
- 登录页和注册页都保留返回按钮，返回目标是患者首页或上一层业务入口。

### 3.5 挂号记录和个人中心

- 挂号记录独立为 `/patient/registers`。
- 未登录时不请求历史接口，只显示登录 / 注册引导。
- 已登录时使用 `GET /api/v1/patient/{patient_uuid}/registers/detail` 请求真实挂号记录。
- 个人中心独立为 `/patient/profile`，承载患者信息、就诊码、常用入口和账户操作，不再混放完整挂号列表。

## 4. 已记录的问题

详细问题记录见 `docs/问题记录.md`。当前已经沉淀的问题包括：

- 注册按钮 `Network Error`：网关不可用和 CORS 预检失败都可能导致，需要同时看浏览器 Network 和后端日志。
- 科室显示 `SJWK`：展示层不能直接显示内部编码，必须转换成中文科室名。
- AI 是否真实接入不清楚：页面必须展示 `source`、`model`、`confidence`，不能把 mock/fallback 说成真实大模型。
- 医生推荐为空：常见原因是没有未来有效排班，不能用前端假医生填充。
- 登录注册流程不自然：已改为首页登录优先、注册成功直接进入已登录流程。
- 挂号时间段过多：已改为日期 -> 午别 -> 时间段，并显示剩余号源。
- 智能分诊入口位置不合理：已改为挂号流程内可选辅助。
- 患者端页面风格不统一：已开始用共享头部、底部导航和蓝白移动端风格统一。
- Codex / Windows 工具问题：PowerShell 中文显示、sandbox helper、Vite 临时文件权限和 `gh` 未安装都需要记录。

## 5. 当前真实差距

- 患者端主链路已经成型，但报告查询、医院信息、消息中心仍未接真实接口，不应先做大面积假业务。
- 医生端和管理员端还只是基础布局与占位，不应在患者端主链路稳定前铺大量页面。
- AI 分诊在未配置真实 key 时仍可能走 mock/fallback，需要继续在页面和答辩中说明边界。
- 演示数据仍需要固化脚本，避免每次手动补排班。
- 号源过期、实时余号刷新、无号状态、支付异常和队列刷新还需要继续加强。
- 需要补一套从 Docker 到前后端联调的启动检查脚本或说明。

## 6. 下一步建议

优先级一：患者端主链路收口

- 固化一套演示患者、科室、医生、排班、号源数据。
- 补齐挂号确认、支付、候诊页面的加载、错误、无号、支付失败状态。
- 继续用 `impeccable` 检查移动端视觉一致性，重点看 360px 到 430px 宽度。

优先级二：真实联调与演示脚本

- 编写或整理一键演示数据脚本。
- 明确启动顺序：Docker Desktop -> `backend/run_microservices.py` -> `frontend/npm run dev`。
- 写出一条可重复演示路线：张三登录 -> 按科室挂号 -> 可选 AI -> 医生推荐 -> 锁号 -> 支付 -> 候诊 -> 挂号记录。

优先级三：医生端最小工作台

- 从真实挂号队列开始做医生端。
- 先做候诊列表、叫号、开始接诊、AI 病历草稿查看与医生确认。
- 不先做复杂后台 BI 或无接口支撑的展示页。

优先级四：管理员端基础后台

- 先做医生、科室、排班、排班审批和 AI 审计日志。
- 管理员端要保持桌面端后台风格，不复用患者端小程序式视觉。
