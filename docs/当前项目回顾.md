# 当前项目回顾

更新时间：2026-07-07

本文档用于快速回顾 SmartBrainClinic 当前真实状态、已经完成的工作、遇到的问题、接下来怎么做，以及当前工作区未提交改动的大致范围。更完整的总计划见 `docs/项目规划.md`，前端专项计划见 `docs/frontend-plan.md`。

## 1. 项目定位

SmartBrainClinic 是面向门诊流程的 AI 赋能 HIS / 无人 AI 医院实训平台。当前实现路线是 Python FastAPI 微服务后端 + Vue 3 前端，不再按早期 Java / Spring Boot 规划推进。

当前固定演示主线是脑科 / 神经外科门诊，围绕“头痛、恶心、视物模糊、疑似脑部占位”等症状，串联患者建档、AI 分诊、医生推荐、挂号、支付、候诊、医生接诊、检查检验、影像辅助分析、处方和电子病历归档。

## 2. 当前技术状态

后端：

- Gateway：`backend/app/main.py`，端口 `8000`。
- Auth：端口 `8001`。
- Patient：端口 `8002`。
- Medical：端口 `8003`。
- Pharmacy：端口 `8004`。
- Billing：端口 `8005`。
- 基础设施：PostgreSQL + pgvector、Redis、RabbitMQ、Nacos、Docker Compose。

前端：

- 路径：`frontend/`。
- 技术栈：Vue 3 + Vite + Vue Router + Pinia + Axios + Element Plus。
- 当前开发服务地址：`http://localhost:5173`。
- 当前前端结构方向：三端独立登录、独立首页、独立流程，但暂不拆成三个仓库。
- 当前前端重点：在保持患者端主链路稳定的前提下，继续把医生端从最小闭环推进到真实接诊工作台。

项目级协作工具：

- 已新增 `PRODUCT.md`，用于给设计和 agent 提供产品上下文。
- 已新增 `.agents/AGENTS.md`，用于记录后续 agent 接手规则。
- 已安装 `.claude/skills/impeccable/`，用于前端设计和一致性检查。

## 3. 已完成的主要工作

### 3.1 后端与联调基础

- 网关已补 CORS，允许 `http://localhost:5173` 和 `http://127.0.0.1:5173` 访问，解决浏览器注册时的 CORS 型 `Network Error`。
- 后端已有患者建档、患者查询、AI 分诊、科室列表、医生推荐、排班、线上挂号、支付模拟、候诊状态、挂号历史等接口。
- 医生推荐依赖真实科室、真实医生、真实排班和真实号源，不用前端假医生补页面。

### 3.2 患者端页面

已经从占位首页拆成真实页面和路由：

- `/patient`：患者登录。
- `/patient/register`：注册建档。
- `/patient/home`：患者首页。
- `/patient/departments`：按科室挂号入口。
- `/patient/triage`：AI 问诊。
- `/patient/doctors`：医生推荐。
- `/patient/confirm-register`：确认挂号。
- `/patient/payment`：线上支付。
- `/patient/queue`：候诊状态。
- `/patient/registers`：挂号记录。
- `/patient/profile`：个人中心。

### 3.3 三端结构决策

- 当前已经明确，产品结构从“患者端独立、医生端和管理员端共享 staff shell”调整为“三端独立”。
- 三端独立的含义是：
  - 患者端独立登录、独立首页、独立流程。
  - 医生端独立登录、独立工作台、独立业务流程。
  - 管理员端独立登录、独立控制台、独立管理流程。
- 当前仍保持一个 `frontend/` 工程，通过 `/patient/*`、`/doctor/*`、`/admin/*` 组织三套路由和页面，不拆成三个前端仓库。
- `frontend/src/App.vue` 顶部全局角色切换按钮已移除，前端不再保留共享产品入口。

### 3.4 医生端当前进度

- 医生端已经不再是占位壳，现有可用入口为：
  - `/doctor/login`
  - `/doctor/workbench`
  - `/doctor/encounter/:registerId`
- 医生登录页当前通过真实医生目录承载身份，登录后将 `employee_uuid` 写入 `doctorSession`。
- 医生工作台已接真实候诊队列，支持：
  - 查询医生当日候诊列表
  - 叫下一位
  - 开始接诊
  - 对接诊中患者继续进入详情
- 接诊详情页已接入真实后端能力：
  - 挂号详情读取
  - AI 病历草稿读取
  - 医生确认病历草稿
  - 相似病历召回
  - AI 医生助手问答
- 当前这条链路已经满足“AI 给建议，医生确认后生效”的最小答辩要求，但还未进入检查、检验、处置、处方开立。

### 3.5 患者端交互与设计

- 患者端已转向蓝白医疗移动端风格，首页、挂号记录、个人中心和主链路页面逐步统一。
- 首页已经不再单独放“智能分诊”入口，智能分诊集成在按科室挂号流程中。
- 从首页进入按科室挂号时，自动弹出“是否需要智能分诊帮助”的提示；从 AI 问诊返回后不重复弹。
- 挂号页顶部提示条可以直接进入 AI 问诊。
- 挂号链路统一采用左侧返回业务上一级、右侧一键返回患者首页的头部结构。
- 底部导航第二项已固定为“挂号记录”，进入 `/patient/registers`。
- 患者端不再展示 `SJWK` 等内部科室编码，展示层使用中文科室名。

### 3.6 登录注册

- `/patient` 是患者登录页，使用姓名 + 证件号查询真实患者档案。
- `/patient/register` 使用真实建档接口。
- 注册成功后直接进入已登录患者流程。
- 身份证输入会自动匹配出生日期，并校验身份证出生日期是否合法。
- 登录页和注册页都保留返回按钮，返回目标是患者首页或上一层业务入口。

### 3.7 AI 问诊真实接入调整

- 患者端 AI 问诊页已去掉“离线兜底 / mock 提示型状态块”。
- 后端当前按“真实 AI 配置优先”工作：未配置 `LLM_API_KEY` 时，AI 分诊接口应直接返回不可用错误，而不是继续让前端把它当作正常在线 AI。
- 当前预留的模型接入方向为 OpenAI 兼容接口，计划使用 SiliconFlow 的兼容入口，模型暂定 `deepseek-ai/DeepSeek-V4-Pro`。
- 下一窗口继续前，应由使用者在后端 `.env` 中自行填写真实 key。

### 3.8 挂号记录和个人中心

- 挂号记录独立为 `/patient/registers`。
- 未登录时不请求历史接口，只显示登录 / 注册引导。
- 已登录时使用 `GET /api/v1/patient/{patient_uuid}/registers/detail` 请求真实挂号记录。
- 个人中心独立为 `/patient/profile`，承载患者信息、就诊码、常用入口和账户操作，不再混放完整挂号列表。

## 5. 已记录的问题

详细问题记录见 `docs/问题记录.md`。当前已经沉淀的问题包括：

- 注册按钮 `Network Error`：网关不可用和 CORS 预检失败都可能导致，需要同时看浏览器 Network 和后端日志。
- 科室显示 `SJWK`：展示层不能直接显示内部编码，必须转换成中文科室名。
- AI 是否真实接入不清楚：页面必须展示 `source`、`model`、`confidence`，不能把 mock/fallback 说成真实大模型。
- 医生推荐为空：常见原因是没有未来有效排班，不能用前端假医生填充。
- 登录注册流程不自然：已改为首页登录优先、注册成功直接进入已登录流程。
- 挂号时间段过多：已改为日期 -> 午别 -> 时间段，并显示剩余号源。
- 智能分诊入口位置不合理：已改为挂号流程内可选辅助。
- 患者端页面风格不统一：已开始用共享头部、底部导航和蓝白移动端风格统一。
- Codex / Windows 工具问题：PowerShell 中文显示、sandbox helper、Vite 临时文件权限，以及 PowerShell 到 Python 标准输入的中文转问号问题都需要持续记录。

## 6. 当前真实差距

- 患者端主链路已经成型，但报告查询、医院信息、消息中心仍未接真实接口，不应先做大面积假业务。
- 医生端已经有独立登录页、独立路由守卫和最小业务闭环，但管理员端还只有骨架。
- 三端独立方向已经明确，但协作文档必须持续同步，避免下个窗口又回到共享 staff shell 思路。
- 真实 AI 接入需要使用者填写后端 key；在 key 未配置前，患者端 AI 问诊无法当作完整演示闭环使用。
- 演示数据已经开始固化，但还缺一条患者端到医生端再到管理员端的正式联动演示脚本。
- 号源过期、实时余号刷新、无号状态、支付异常和队列刷新还需要继续加强。
- 需要补一套从 Docker 到前后端联调的启动检查脚本或说明。
- 医生端接诊流程已经能落地病历确认，但检查、检验、处置、处方等后续动作仍未接入。
- 队列刷新目前仍偏手动刷新，尚未正式接入实时更新策略。

## 7. 下一步建议

建议不要再按“想到哪个页面就补哪个页面”的方式推进。当前三端会话与医生端最小闭环已经到位，后续顺序建议固定为：先把医生端接诊动作继续做深，再补管理员端真实控制台，最后收口演示路线与答辩材料。

优先级一：医生端接诊页继续做深

- 在 `/doctor/encounter/:registerId` 中补检查 / 检验 / 处置开单。
- 让医生能看到已开单项目和其状态，而不只停留在病历确认。
- 评估是否把处方推荐也纳入这一页，至少先打通一类真实动作。

优先级二：医生端稳定性与回归验证

- 为工作台补更稳定的自动刷新策略，优先考虑复用既有队列事件 / WebSocket。
- 做一轮可复述的医生端回归：登录 -> 候诊 -> 开始接诊 -> 病历确认 -> 队列状态变化。

优先级三：管理员端基础后台

- 新增 `/admin/login` 与 `/admin/console`。
- 先做医生、科室、排班、排班审批和 AI 审计日志。
- 管理员端要保持桌面端后台风格，不复用患者端小程序式视觉。

优先级四：真实联调与演示脚本

- 继续整理一键演示数据脚本和启动说明。
- 明确启动顺序：Docker Desktop -> `backend/run_microservices.py` -> `frontend/npm run dev`。
- 写出一条可重复演示路线：患者登录/挂号 -> 候诊 -> 医生接诊确认 -> 管理员查看排班或审计。
