# GitHub 协作规范

本规范以当前仓库的实际协作方式为准：`main` 是稳定主线，不额外维护长期 `develop` 分支。是否使用功能分支和 Pull Request（PR），由改动风险与协作场景决定。

## 1. 基本原则

- 每次提交只处理一个明确目标，避免混入无关改动。
- 修改前先确认工作区状态，不覆盖他人或尚未提交的改动。
- 提交前检查 diff，并完成与改动范围匹配的验证。
- 未经明确要求不执行 `git push`，不使用 force push，不重写历史。
- 涉及路由、接口、数据库或产品边界时，同步更新对应文档。

## 2. 分支策略

- `main`：当前稳定主线，只保留经过验证的提交。
- 小型、低风险且已明确约定的改动，可以在 `main` 上形成原子提交。
- 高风险改动、多人并行开发或需要评审的功能，应创建短期分支并通过 PR 合并。
- 分支名应表达目的，例如 `feature/doctor-ai-context`、`fix/register-cache` 或 `codex/docs-consolidation`。
- 分支合并后及时删除，不建立无实际维护需求的长期集成分支。

## 3. Commit 规范

推荐格式：

```text
<type>(<scope>): <中文简述>
```

常用类型：

- `feat`：新增功能。
- `fix`：修复缺陷。
- `docs`：文档更新。
- `refactor`：不改变外部行为的重构。
- `test`：测试补充或调整。
- `chore`：依赖、配置或工程维护。

示例：

```text
feat(doctor): 增加接诊上下文展示
fix(patient): 修复挂号记录缓存失效
docs: 收束项目文档体系
```

## 4. 提交前验证

根据变更范围选择验证项，不要求无关模块重复执行：

- 前端：在 `frontend/` 下执行 `npm run build`，并回归关键页面。
- 后端：执行相关 `pytest`，必要时补充 `py_compile` 或服务启动检查。
- 数据库：确认迁移脚本可执行，并检查旧数据兼容性。
- 文档：检查本地链接、命令、路由、接口名称和当前代码是否一致。
- 所有改动：执行 `git diff --check`，确认没有意外文件进入提交。

## 5. Pull Request 要求

使用 PR 时，应包含以下信息：

- 变更目标及用户可见影响。
- 核心实现和必要的架构取舍。
- 已执行的验证及结果。
- 数据迁移、兼容性或剩余风险。
- 医疗结论、AI 建议、处方等功能必须说明医生确认和审计边界。

## 6. 发布与安全

- 推送或发布前再次确认目标分支、远端地址和待提交文件。
- 不提交密钥、令牌、真实患者信息或本地环境凭据。
- 数据库连接和第三方服务配置通过环境变量管理。
- 出现验证失败时先修复或记录阻塞原因，不以关闭安全检查换取通过。
