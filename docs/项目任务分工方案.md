# SmartBrainClinic 项目任务分工方案

更新时间：2026-07-14
适用范围：当前一期交付与答辩闭环，不重新划分已经完成的历史代码归属。

## 1. 分工目标与依据

项目当前是面向脑科/神经外科门诊的 FastAPI 微服务与 Vue 3 前端，已具备患者、医生、管理员三端入口。患者挂号主链已形成，后续交付重点是医生接诊闭环、CT 伪影辅助分析、管理员真实后台，以及三者之间可演示、可验证的贯通流程。

本方案按**功能模块端到端负责**，而不是按“前端/后端”切割。每位成员对其模块中的页面、接口、服务逻辑、数据库 migration、自动化测试、模块文档共同负责；模块间的契约、集成、回归和发布由项目负责人统一负责。

| 成员 | 占比 | 定位 |
| --- | ---: | --- |
| 你（项目负责人） | 65% | 全部 AI 与 CT 影像能力 owner；医生核心闭环、跨模块集成、验收和发布负责人 |
| 成员 A | 15% | 非 AI 患者服务与就诊体验 owner |
| 成员 B | 10% | 非 AI 管理治理、账号与运营配置 owner |
| 成员 C | 10% | 非 AI 排班、收费与药房履约 owner |
| **合计** | **100%** |  |

## 2. 模块分工与工作量

### 2.1 你（65%）：核心诊疗闭环与系统集成

#### A. 医生接诊、病历和医嘱闭环（20%）

- 前端：负责 `DoctorWorkbenchView.vue`、`DoctorEncounterView.vue` 及医生会话/路由相关改造；完善今日队列、叫号、开始/继续接诊、病历确认、统一开单、已开项目与处方入口。
- 后端：负责 Patient 与 Medical 服务中接诊状态流转、检查/检验/处置统一签署、结果录入和医生侧查询接口；需要时协调 Pharmacy 的处方契约。
- 数据与验证：维护医疗记录、检查检验处置相关 migration 与契约测试；保证“候诊 → 接诊 → 病历/开单 → 结果”可以从医生端完成演示。
- 主要边界：不负责药房发药、库存扣减和账单退款的内部实现，但负责在接诊页正确触发和展示其公开结果。

#### B. 全部 AI 能力、CT 伪影分析与报告查验闭环（25%）

- AI 范围：统一负责患者 AI 分诊、医生推荐、挂号 AI 上下文、病历草稿辅助、AI 处方推荐、AI 审计规则，以及所有模型的降级提示、人工确认和结果可追溯性。
- 前端：负责医生端 AI 分诊上下文、病历草稿辅助、AI 处方建议，以及 CT 影像输入选择、任务状态、叠加预览/交互切片、检查报告草稿、审核发布和更正入口；患者端只展示 AI 的非诊断性引导结果和已发布报告。
- 后端与模型服务：负责 AI 服务调用、结果校验、审计记录、Medical 服务 CT 任务契约、受控影像引用、CTArtifact 调用、任务状态、预览与切片代理、报告版本控制；维护 `model_services/ct_artifact` 与统一启动脚本的可运行性。
- 安全边界：患者端不得读取草稿、模型权重、本机绝对路径、原始 DICOM 或内部推理统计；模型结果只能辅助医生，不能自动写入正式诊断结论。
- 验证：覆盖任务提交/查询、失败可见性、报告发布与更正链路；在真实本地模型环境补做服务存活和图像渲染回归。

#### C. 跨模块拼接与集成验收（12%）

- 制定并冻结跨域 API 契约、状态机和错误语义：患者挂号、排班号源、支付、候诊、医生接诊、医嘱、收费、报告发布之间的数据交接由你统一确认。
- 负责 Gateway 路由、共享类型/请求层、会话守卫、演示数据、跨服务问题定位和冲突处理；不替代各模块 owner 修复其模块内缺陷。
- 负责端到端演示主线：`患者建档/挂号 → 支付 → 候诊 → 医生接诊/开单 → 缴纳后续费用 → 报告发布 → 患者查询`。

#### D. 质量门禁、发布与答辩集成（8%）

- 建立集成分支的合并准入：代码评审、migration 顺序、环境配置、测试回归、演示脚本和风险清单。
- 统一执行全链路测试、前端构建、后端编译/定向 pytest、启动健康检查和关键页面人工验收；维护 `README.md`、`docs/frontend-plan.md` 与项目级演示说明的最终一致性。
- 负责最终发布、版本标签（如需）和答辩时的主流程串讲。

### 2.2 成员 A（15%）：非 AI 患者就诊服务闭环

- 前端：负责患者端的登录/建档、科室选择、挂号确认、挂号支付、候诊、历史挂号、缴费中心、报告查询、个人中心、消息/医院信息等页面的可用性、移动端适配与错误/空态。
- 后端：负责 Patient 服务中患者资料、非 AI 挂号流程、候诊查询、患者侧待缴项目/缴费记录和已发布报告聚合接口的模块内实现与测试；AI 分诊、医生推荐和 AI 上下文的业务逻辑由你提供稳定契约后接入。
- 数据与文档：维护患者侧 migration、种子数据、接口契约测试和患者流程文档；明确概念展示页与真实业务接口的边界。
- 交付验收：能独立证明“登录/建档 → 挂号 → 支付 → 候诊”和“患者仅查看已发布报告”成立，再交给你做跨模块联调。
- 不负责：AI 分诊、医生推荐、AI 上下文、医生接诊状态机、Medical 报告发布实现、Billing/Pharmacy 内部扣费与退款实现。

### 2.3 成员 B（10%）：管理员账号、基础治理与运营配置

- 前端：负责管理员登录、驾驶舱、账号管理、科室/诊室等基础资料管理、审批中心和运营统计页；保持 `AdminLayout`、会话守卫、列表分页/筛选/加载失败状态的一致性。
- 后端：负责 Auth 服务的管理员认证、管理员/医生账号与基础资料管理、资源统计，以及非 AI 审批动作的接口与授权校验。
- 数据与验证：维护管理员账号、权限、基础资料和运营配置相关 migration、初始化脚本、权限边界测试；确保未登录或权限不匹配的请求不能访问管理员数据。
- 对接边界：排班、药房、账单页面的数据源由成员 C 提供；成员 B 负责把它们挂入统一管理员壳和治理视图，但不改写其业务服务。

### 2.4 成员 C（10%）：排班、收费与药房履约

- 前端：负责管理员侧排班、账单、药房工作台页面，以及与患者缴费中心/医生处方入口所需的最小展示与操作契约；不接管患者端总体视觉与路由。
- 后端：负责排班规则、排班申请与审批结果落库、号源生成；负责 Billing 的合并支付、账单查询、退款保护；负责 Pharmacy 的非 AI 处方执行、库存、发药/退药和药房工作台接口。
- 数据与验证：维护排班、账单、药品库存相关 migration、幂等/库存/退款边界测试和演示数据；保证支付、退款、发药不会重复执行或造成库存负数。
- 对接边界：AI 处方推荐的算法、调用和结果校验由你负责；成员 C 负责将经医生确认的处方执行为收费、库存和发药结果，并以明确接口交给成员 A、你和成员 B。跨服务最终串联由你负责。

## 3. 模块交接契约

| 上游模块 | 下游模块 | 必须交付的契约 | 最终验收人 |
| --- | --- | --- | --- |
| 成员 A：患者挂号 | 成员 C：排班/收费 | 挂号单、排班号源、支付状态、待缴项目的稳定 ID 与状态枚举 | 你 |
| 成员 C：支付/药房 | 你：医生接诊 | 已支付状态、处方/医嘱对应的收费与发药状态 | 你 |
| 你：检查与报告 | 成员 A：患者查询 | 仅已发布的报告摘要、版本、发布时间；不泄露草稿和模型内部数据 | 你 |
| 成员 C：排班/账单/药房 | 成员 B：管理员 | 分页查询、审批动作、库存和账单汇总的管理员授权接口 | 成员 B + 你 |
| 成员 B：账号与权限 | 全部模块 | 管理员/医生身份、权限失败语义和演示初始化方式 | 你 |
| 你：AI 与 CT | 成员 A/B/C | AI 结果的展示契约、人工确认要求、审计记录和失败降级语义 | 你 |

## 4. 协作规则

1. **每个需求只设一个模块 owner。** 其他成员只能通过 issue、接口约定或 code review 协作，避免多人同时修改同一服务文件。
2. **前后端一起交付。** 任何页面改动必须同时给出 API 依赖、加载/空/错误状态、测试证据和对应文档更新；任何新增接口必须给出调用页面或明确标为预留接口。
3. **先契约、后联调。** 跨模块接口先在文档或 PR 描述中写清请求、响应、状态枚举、鉴权和失败语义，再开始各自实现。
4. **保持小而可合并的提交。** 一个提交只覆盖一个可验证切片；不得夹带 `outputs/` 截图、个人实验文件或其他成员的未完成改动。
5. **交付给你之前必须自验。** 前端至少完成类型检查/构建，后端至少完成相关 pytest 与 migration 可执行性检查；未能执行的验证必须如实说明。
6. **集成只由你合并。** 你负责处理冲突、运行端到端回归、确认文档一致性和决定是否进入演示版本。

## 5. 推荐实施顺序

1. **先冻结基础契约（你主导）**：账号身份、挂号状态、支付状态、检查/报告状态和管理员授权边界。
2. **并行完成各自模块内闭环**：成员 A 完善非 AI 患者主链，成员 B 完成账号/基础治理，成员 C 完成排班/收费/药房履约，你推进全部 AI 能力、医生接诊与 CT/报告。
3. **按业务流两次集成**：先打通“挂号—支付—候诊—接诊”，再打通“开单—后续缴费/发药—报告发布—患者查询”。
4. **最终收口（你负责）**：回归测试、演示数据、异常路径、服务启动、文档和答辩脚本。

## 6. 当前项目状态下的风险提示

- 当前工作区存在尚未提交的 CT 伪影服务、Medical 接口和医生接诊页改动。该切片归你所有；其他成员不要直接修改这些文件，先通过契约对接。
- 患者端已有较完整主链，成员 A 的工作应以真实接口、状态恢复、错误处理和回归为主；AI 分诊和推荐能力只负责接入展示，不由成员 A 维护算法或服务逻辑。
- 管理员端已有最小页面和部分接口，但完整运营链路尚未闭合；成员 B 与 C 的边界以“账号/基础治理”和“排班/账单/药房业务”划分。AI 审计规则、模型结果和高风险 AI 复核逻辑归你负责。
- CT 模型服务依赖本地 Python/GPU 环境，最终验收要把“代码测试通过”和“真实模型服务可运行”分开记录。
