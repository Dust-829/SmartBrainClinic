更新时间：2026-07-14。

本文对应 [backend/scripts/seed_full_demo_data.sql](D:/work/SmartBrainClinic/backend/scripts/seed_full_demo_data.sql)，用于补齐患者端、医生端和后台相关流程的相对日期演示数据。它不是旧版医生工作台最小样例的替代说明，而是当前推荐的主演示数据入口。

## 1. 使用前提

- 目标库已经导入 `backend/his_db_backup.sql`。
- 已按顺序执行 `backend/migrations/*.sql`。
- 尤其要先执行 `backend/migrations/20260708_01_create_ai_conversation_tables.sql`，否则脚本会因为缺少 `ai_conversation_session` / `ai_conversation_message` 而直接失败。
- 如需演示管理员端真实登录和账号管理接口，还必须执行 `backend/migrations/20260714_02_create_admin_account_auth.sql`；该迁移不由 `seed_full_demo_data.sql` 替代。

## 2. 脚本入口

当前仅保留 `backend/scripts/seed_full_demo_data.sql` 作为演示数据入口。

- 它覆盖患者端、医生端、AI 上下文、医技、收费和异常工单的全链路样例。
- 如果要验收“数据足量、页面不空、链路互相关联”，应直接使用这份脚本。
- 后续新增页面依赖字段时，应继续扩充 `seed_full_demo_data.sql`，不再恢复旧的最小样例脚本。

## 4. 日期策略

脚本不再写死 `2026-07-07` 这类历史日期，而是统一基于 `CURRENT_DATE` 生成数据：

- `-1` 天：历史已完成病例
- `0` 天：今天的候诊与接诊
- `+1` 天：明天的预约与异常工单
- `+2` 天：后天的复诊与改期样例

这样在任意执行日重跑脚本，`GET /api/v1/patient/doctor/{employee_uuid}/queue`、挂号记录、可预约排班和异常工单都仍然有意义，不会因为日期过期而整页变空。

### 5.3 管理员认证账号

“周院管”是演示业务数据中的 `Employee`，用于排班审核、运营协同等业务展示；它不是管理员端登录账号。管理员端使用独立的 `AdminAccount` 身份，不能用医生或员工的默认密码登录。

管理员认证数据不写入 `seed_full_demo_data.sql`，按以下方式初始化：

1. 在 `backend/.env` 配置 `JWT_SECRET_KEY`、`ADMIN_BOOTSTRAP_STAFF_CODE`、`ADMIN_BOOTSTRAP_DISPLAY_NAME`、`ADMIN_BOOTSTRAP_PASSWORD`。
2. 在 `backend/` 目录执行 `python scripts/bootstrap_admin.py`。
3. 使用该管理员凭据请求 `POST /api/v1/auth/admin/login`，成功后得到短期 `access_token`。

本机开发环境已完成数据库迁移和 bootstrap 初始化，因此存在可成功登录管理员端的账号。登录时填写本机 `backend/.env` 中的 `ADMIN_BOOTSTRAP_STAFF_CODE` 与 `ADMIN_BOOTSTRAP_PASSWORD`；这两个字段分别就是管理员端的账号和密码。若换了数据库、清空了 `admin_account` 表，或这两个字段为空，应重新按上述步骤配置并运行 bootstrap，而不是尝试使用“周院管”或医生默认密码。

真实密码、JWT 密钥和 access token 均只保存在本机受控环境；不得写入 SQL、仓库、本文档、截图或录屏。

## 6. 管理员端演示数据与验收

管理员端演示依赖两部分：本清单脚本提供医生、患者等业务数据；上一节的 bootstrap 脚本提供独立管理员身份。两者均准备完成后，可按以下顺序验收：

1. 不携带 Token 请求 `GET /api/v1/patient/admin/patients?limit=1`，应返回 `401`。
2. 使用初始化的管理员账号调用 `POST /api/v1/auth/admin/login`，应返回 `access_token`。
3. 使用 `Authorization: Bearer <access_token>` 调用 `GET /api/v1/auth/admin/doctors?limit=1&offset=0`，应返回 `data.items` 与 `data.pagination`；`pagination` 包含 `total`、`limit`、`offset`。
4. 修改 `offset`（例如 `offset=1`）可演示分页请求。若数据总数不超过单页条数，后续页为空是正常结果。
5. 打开 `http://localhost:5173/admin/login`，登录后进入 `/admin/accounts`，医生和患者两个标签页都应能加载首屏数据。

当前 M2 只完成了服务端分页协议和首屏接入；页码、上一页/下一页和每页条数控件属于待实施的 M3，演示时不应表述为已完成。

## 7. 验收重点

执行脚本后，至少应满足：

1. 今天至少 2 位医生的工作台队列非空。
2. 患者挂号记录中同时存在历史、今日、未来、取消和待支付样例。
3. `/api/v1/patient/register/{register_uuid}/ai-context` 对部分挂号可返回真实分诊摘要和消息。
4. `/doctor/encounter/:registerId` 至少能看到：
   - 一个已确认病历样例
   - 一个草稿待确认样例
   - 一个检查/检验/处置队列样例
5. 账单接口能返回已收费与已退费混合结果。
6. 患者异常工单页不是空白。
7. 管理员未登录访问患者档案管理接口返回 `401`；登录后可获取受保护的医生列表及分页结构。

## 8. 关系边界

这份全链路样例的设计重点不是“随机造很多行”，而是保证几条主链真实闭环：

- `register` 是核心中枢。
- `scheduling_actual.registered_count` 与 `scheduling_time_slot.is_booked` 按当前挂号状态回写。
- `medical_record`、`ai_conversation_session`、`check_request`、`prescription`、`outpatient_bill` 都围绕真实 `register_uuid` 关联。
- 已取消挂号会保留历史记录，但不再占用未来号源。
- 今日医生队列只包含 `REGISTERED` 与 `RECEPTION`。
- `AdminAccount` 与 `Employee` 是两套身份模型：前者用于管理员认证和账号管理授权，后者用于医生/员工业务数据；演示时不得混用两类凭据。

## 9. 已知边界

- 这份脚本会补足“可演示的数据闭环”，但不会替代真实线上业务流转。
- 如果需要验证收费、退费、改签等状态机的运行逻辑，仍应以接口实际调用为准，而不是只看静态库数据。
- 若后续再新增页面依赖字段，应优先扩充 `seed_full_demo_data.sql`。
- 管理员 bootstrap 只负责创建受控的认证身份，不应把真实凭据塞入演示 SQL；若需重置管理员账号，应在受控环境中显式处理，而非修改种子数据。
