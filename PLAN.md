## Goal

基于现有医生队列接口，将医生工作台顶部替换为“医生身份 + 挂号状态环图 + 分时段挂号条形图”，并完成刷新、空状态、响应式和验收收口。

## Current behavior

- 医生工作台顶部已经是“医生身份 + 状态环图 + 分时段挂号条形图”三栏布局。
- 工作台真实数据来源仅为 `GET /api/v1/patient/doctor/{employee_uuid}/queue` 返回的候诊数组。
- 前端已稳定推导今日队列总数、状态分布和时段分布，并继续提供当前接诊、下一位和最近叫号信息。
- 项目未安装专用图表库，环图和条形图使用医生端局部 SVG / CSS 组件实现。
- 主文档中仍残留少量旧口径，与当前代码已实现的医生接诊页能力不完全一致。

## Proposed solution

- 已生成高保真设计图与 v2 实施计划。
- 已完成状态聚合、SVG 环图、CSS 条形图、三栏布局和 15 秒刷新状态。
- 已完成首次加载、失败状态、空队列、过期响应保护和窄屏适配。
- 阶段 B 的等待时长、异常预警和趋势统计继续等待聚合接口，不在前端伪造。

## Risks

- 如果设计稿直接假设“已完成数、平均等待时长、时段吞吐”等统计存在，会与当前接口现实脱节。
- 如果为了可视化直接引入新图表库，会扩大改动面，不符合当前仓库“小步收口”的方式。
- 如果只补设计稿、不修正文档旧状态，后续实现阶段仍可能被过时计划误导。

## Validation strategy

- 逐项比对 `frontend/src/views/doctor/DoctorWorkbenchView.vue`、`frontend/src/api/doctor.ts`、`backend/app/microservices/patient/services/patient_service.py` 的真实字段与行为。
- 比对 `docs/frontend-plan.md`、`docs/项目规划.md` 当前口径与实际代码是否一致。
- 确认设计计划明确区分：
  - 当前接口可直接支持的可视化
  - 需要新增接口后才能补齐的高级统计

- 当前本地 8 位医生今日队列均为空，非空真实队列场景需在出现当日挂号后补验。
